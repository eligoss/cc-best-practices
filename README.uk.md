# Найкращі практики Claude Code

[English](.) | [Русский](ru) | **Українська** | [Español](es)

> **Для кого це?** Для розробників, які вже використовують Claude Code і хочуть перейти від "воно якось працює" до "це справді у 10 разів продуктивніше." Незалежно від того, чи ти користуєшся ним тиждень, чи пів року — тут є щось для тебе.

Більшість людей встановлюють Claude Code, вводять промпти і отримують непогані результати. Але є колосальна різниця між тим, щоб використовувати його поверхово, і тим, щоб налаштувати так, щоб Claude справді *розумів* твої проєкти, пам'ятав твої рішення і працював у твоєму стилі. Цей гайд заповнює цю прогалину.

Почнемо з базових блоків — plugins, MCP servers і команди — а потім розберемо практики, що накопичують ефект з часом: як структурувати інструкції, коли який інструмент використовувати, і як зробити так, щоб сесії не починалися щоразу з нуля.

## 1. Екосистема розширень

Claude Code з коробки — це добре. Claude Code з правильними розширеннями — це зовсім інший досвід.

Є три способи його розширити, і вони працюють разом:

### Plugins — головне

Plugins об'єднують усе: slash-команди, спеціалізованих агентів, hooks і іноді MCP tools. Уяви їх як "пакети функцій" для Claude Code.

```bash
# Переглянь маркетплейс плагінів
/plugins

# Або встанови напряму
claude plugin install superpowers@claude-plugins-official
```

Ось plugins, які варто встановити одразу — про важливіші з них ми поговоримо детальніше далі:

| Plugin | Навіщо він тобі |
|--------|----------------|
| **superpowers** | Планування, brainstorming, TDD, debugging, worktrees, code review — структуровані workflow, що роблять Claude дисциплінованим |
| **claude-md-management** | AI-допомога у написанні та аудиті CLAUDE.md |
| **skill-creator** | Перетворює будь-які знання про проєкт у багаторазовий skill |
| **serena** | Семантична навігація по коду — знаходить символи, references, редагує за змістом |
| **context7** | Жива документація бібліотек, щоб Claude не вигадував застарілі API |
| **code-review** | PR review workflows з паралельними subagent reviewers |
| **coderabbit** | Інтеграція з [CodeRabbit](https://www.coderabbit.ai/) AI review |
| **feature-dev** | Guided feature development з дослідженням кодобази |
| **playwright** | Автоматизація браузера і тестування |
| **plugin-dev** | Інструменти для створення власних plugins |

### MCP Servers — зовнішні інструменти

MCP (Model Context Protocol) servers з'єднують Claude із зовнішніми системами — твоєю IDE, трекером завдань, графом знань, інструментами моніторингу. Вони надають сирі інструменти, які Claude викликає напряму.

```bash
claude mcp add <name> -- <command>
claude mcp list
```

Кілька часто використовуваних:

| Server | Що робить |
|--------|-------------|
| **Graphiti** | Граф знань для постійної пам'яті |
| **JetBrains** | Збірка, тестування і рефакторинг з IDE |
| **Atlassian** | Jira + Confluence (часто попередньо встановлений через OAuth) |
| **Sentry** | Відстеження помилок і дослідження проблем |
| **Docker MCP** | Каталог MCP servers, які можна встановити за потребою |

### Slash Commands — інтерфейс користувача

Slash commands — це як ти викликаєш skills з plugins. Набери `/` і побачиш, що доступно:

```text
/commit              Створити git commit
/code-review         Зробити review pull request
/revise-claude-md    Оновити CLAUDE.md з результатами сесії
/skill-creator       Створити новий skill із знань
/simplify            Переглянути і спростити змінений код
```

Ось короткий огляд. Тепер розберемо те, що реально має значення.

## 2. CLAUDE.md — Твої інструкції, що залишаються

Якщо ти зробиш лише одну річ з цього гайду, нехай це буде ця: **напиши хороший CLAUDE.md.**

Щоразу на початку сесії Claude читає твої CLAUDE.md файли. Це твої постійні інструкції — те, що ти б інакше повторював у кожному діалозі. Правильно розподіли глобальний і проєктний файли, і Claude розпочинатиме кожну сесію, вже знаючи як ти працюєш.

### Два файли, дві ролі

| | Глобальний (`~/.claude/CLAUDE.md`) | Проектний (`<repo>/CLAUDE.md`) |
|-|-------------------------------|------------------------------|
| **Завантажується** | Кожну сесію, кожен проєкт | Тільки в цьому проєкті |
| **Думай про нього як** | Твій профіль розробника | Довідник по проєкту |
| **Містить** | Як *ти* працюєш | Як *ця кодова база* влаштована |

### Твій глобальний файл: профіль розробника

Це розказує Claude про тебе у всіх проєктах. Напиши один раз, вдосконалюй з часом:

- **Які інструменти доступні** і коли активувати кожен
- **Як тобі подобається форматувати commits** і яка стратегія гілок
- **Твоя планка якості коду** — DRY, SOLID, строга типізація, що тобі важливо
- **Наскільки ретельно перевіряти роботу** перед тим як сказати "готово"
- **Яка AI-модель для якого завдання** — Haiku для пошуку, Opus для архітектури
- **Твої активні проєкти** — імена, шляхи, Jira ключі для швидкого референсу

Ось як виглядає хороший скелет глобального CLAUDE.md:

```markdown
# Global Claude Code Instructions

## Tools & MCP Servers
| Tool | Purpose | When to Activate |
|------|---------|-----------------|
| Serena | Code navigation | activate_project("<name>") at session start |
| Graphiti | Persistent memory | search_memory_facts at session start |
| Context7 | Library docs | Before writing integration code |

## Git
Commits: <type>(<scope>): <description>
Never commit to main. Never force push. PRs require confirmation.

## Code Quality
Read before modifying. Comments explain *why*. No `any` types.

## Subagents
- haiku — exploration, quick lookups
- sonnet — implementation, tests, code review
- opus — architecture, security-critical decisions
```

> **Порада профі:** Цільуйся приблизно на 100-200 рядків. Борис Черний (автор Claude Code) казав: "близько 100 рядків працює краще, ніж 800." Кожен рядок повинен пройти перевірку: *чи призведе видалення цього до помилки Claude?* Якщо Claude робить правильно без правила — видали правило.

### Твій проєктний файл: довідник по кодовій базі

Тут живуть знання, специфічні для проєкту. Думай про нього як про документ онбордингу, який ти б давав новому члену команди:

- **Архітектура** — моделі даних, ключові файли, як все пов'язано
- **Команди для збірки/тестування/linting** — точні команди, без вгадування
- **Стандарти коду, специфічні для цього проєкту** — "використовуй `Logger`, не `print()`"
- **Типові підводні камені** — те, що кусає саме *в цій* кодовій базі
- **Інтеграційні вказівники** — Jira ключ, Graphiti `group_id`, ім'я проєкту Serena

> **Що тут не місце:** Загальні поради з кодування (це глобальний файл), інструкції з активації інструментів (теж глобальний), будь-що тимчасове (це для tasks або memory).

### Починаєш з нуля? Використай `/init`

Якщо ти налаштовуєш CLAUDE.md для існуючого проєкту, не починай з порожнього файлу. Команда `/init` у Claude Code досліджує твою кодову базу — package files, конфіг, структуру директорій, існуючу документацію — і автоматично генерує стартовий CLAUDE.md:

```bash
# Інтерактивний багатоетапний процес, який спочатку вивчає, потім пише
CLAUDE_CODE_NEW_INIT=1 claude /init
```

Це хороша відправна точка. Потім вдосконалюй з часом за допомогою `/revise-claude-md` (детальніше нижче).

### Правила для шляхів: завантажуй потрібний контекст для потрібних файлів

> *Я ще не використовував цього сам, але це надто корисно, щоб не порекомендувати.*

Замість того, щоб завантажувати всі правила весь час, ти можеш прив'язати правила до конкретних патернів файлів за допомогою `.claude/rules/`:

```yaml
# .claude/rules/api-conventions.md
---
paths:
  - "src/api/**/*.ts"
---

## API Design Rules
- All endpoints return { data, error, meta }
- Use Zod for request validation
- Rate limit: 100 req/min per user
```

Цей файл завантажується лише коли Claude працює з файлами, що відповідають `src/api/**/*.ts`. Твої правила для тестів завантажуються лише при редагуванні тестів. Твої патерни для фронтенду — лише при роботі з компонентами. Та сама ідея, що й skills (Розділ 3), але тригером є шлях до файлу, а не завдання.

### Не пиши сам

У plugin **claude-md-management** є два skills, що роблять важку роботу:

```bash
claude plugin install claude-md-management@claude-plugins-official
```

| Команда | Що робить |
|---------|-------------|
| `/claude-md-improver` | Аудитує твій існуючий CLAUDE.md на прогалини, суперечності і пропущені найкращі практики |
| `/revise-claude-md` | Після робочої сесії фіксує те, що ти дізнався — нові патерни, підводні камені, виправлення |

**Звичка, що окупається:** Запускай `/revise-claude-md` наприкінці будь-якої сесії, де ти дізнався щось нове про проєкт. З часом твій CLAUDE.md еволюціонує зі стаба у вичерпний, перевірений в бою довідник — без того, щоб писати його з нуля.

### Успадкування CLAUDE.md (монорепо і вкладені директорії)

Claude Code проходить вверх по дереву директорій і завантажує кожен знайдений CLAUDE.md:
- `~/.claude/CLAUDE.md` — твій глобальний профіль
- `./CLAUDE.md` або `./.claude/CLAUDE.md` — корінь проєкту (комітти в git для команди)
- Файли CLAUDE.md у дочірніх директоріях — завантажуються ліниво, коли Claude працює в них

У монорепо це означає, що кореневий CLAUDE.md охоплює спільні конвенції, а кожна директорія пакету може мати свої перевизначення. Використовуй `claudeMdExcludes` у settings, якщо файли інших команд створюють шум.

## 3. Skills — Завантажуй знання коли потрібно

Ось проблема, на яку ти швидко натрапиш: твій CLAUDE.md стає довгим. Ти постійно додаєш правила — API конвенції, патерни тестування, стандарти бази даних, чеклисти для деплою — і зрештою кожна сесія завантажує 2000 токенів контексту незалежно від того, потрібні вони чи ні.

Skills вирішують це. Вони схожі на секції CLAUDE.md, що завантажуються **за запитом** замість того, щоб бути завжди.

### Ідея за 30 секунд

Skill — це markdown файл з описом. Коли Claude починає завдання, він перевіряє: "Чи відповідає якийсь skill тому, що я збираюся зробити?" Якщо так — завантажує контент цього skill. Ні — не заважає.

```text
Ти: Виправ flaky тест у UserService

Claude: [бачить, що це завдання з тестування]
        [завантажує skill test-driven-development]
        [завантажує skill testing-conventions твого проєкту]
        [працює з повним контекстом тестування — нічого зайвого не завантажено]
```

### До і після

**До** — все напхано в CLAUDE.md:

```markdown
## API Design Rules
- All endpoints return { data, error, meta }
- Use Zod for request validation
- Rate limit: 100 req/min per user
- Error codes follow RFC 7807
... (ще 50 рядків)

## Database Conventions
- All tables have created_at, updated_at
- Use soft deletes
- Indexes on all foreign keys
... (ще 40 рядків)
```

Кожна сесія завантажує все це. Виправляєш CSS баг? Ти все одно тягнеш API і правила бази даних у контекст.

**Після** — легкий CLAUDE.md, знання за запитом:

```markdown
## Available Skills
- `/api-design` — API conventions, response format, error codes
- `/db-conventions` — Database patterns, migrations, indexes
```

Тепер контекст API завантажується лише коли ти справді працюєш з endpoints. Ті самі знання, менше витрат, кращий фокус.

### Як створити skill

Тобі не потрібно писати skill файли вручну. Plugin **skill-creator** робить це за тебе:

```bash
claude plugin install skill-creator@claude-plugins-official
```

Просто розкажи, що знаєш:

```text
Ти: /skill-creator

Claude: Які знання ти хочеш перетворити на skill?

Ти: Наш API завжди повертає відповіді в такому форматі: { data, error, meta }.
     Ми використовуємо Zod для всієї валідації запитів. Ліміт — 100 req/min.
     Помилки слідують RFC 7807. Всі endpoints версіоновані під /api/v1/...

Claude: [Створює добре структурований skill файл з правильним frontmatter,
         умовами тригеру та організованими інструкціями]
```

Ти надаєш знання, plugin займається форматуванням і описами тригерів.

### Що робить skill файл ефективним

```markdown
---
name: api-design
description: Use when creating or modifying API endpoints — covers response
  format, validation, error handling, and versioning conventions
---

## Response Format
All endpoints return:
{ "data": ..., "error": null | { ... }, "meta": { ... } }

## Validation
Use Zod schemas for all request bodies...
```

Поле `description` робить більшу частину роботи. Воно каже Claude *коли* завантажувати skill, тому пиши його як умову тригеру: "Use when [робиш X] — covers [Y і Z]."

> **Правило: якщо ти пояснював щось Claude більше двох разів в різних сесіях — це має стати skill.**

## 4. Superpowers — Plugin що змінює все

Якщо ти встановиш один plugin з цього гайду — нехай це буде цей.

Ось у чому справа з Claude: він *здатний* на відмінні інженерні практики — планування перед кодингом, написання тестів спочатку, методичний debugging. Але наданий сам собі, він схильний одразу писати код. Ми всі так робимо. Різниця в тому, що Superpowers робить ці практики **автоматичними і обов'язковими**, а не просто бажаними.

```bash
claude plugin install superpowers@claude-plugins-official
```

### Що змінюється після встановлення

Superpowers додає набір **process skills**, що активуються залежно від того, що ти робиш:

| Коли ти... | Superpowers активує... | Що означає: Claude буде... |
|----------------|-------------------------|---------------------------|
| Починаєш щось нове | **Brainstorming** | Пропонувати 2-3 підходи з компромісами перед написанням будь-якого коду |
| Готовий будувати | **Writing Plans** | Створювати покроковий план реалізації для твого затвердження |
| Будуєш | **Executing Plans** | Слідувати плану крок за кроком з відстеженням прогресу |
| Пишеш будь-яку функцію або виправлення | **Test-Driven Development** | Спочатку писати тест, що падає, *потім* реалізацію |
| Натрапляєш на баг | **Systematic Debugging** | Формувати гіпотези, методично їх перевіряти, звужувати причину |
| Завершуєш блок роботи | **Code Review** | Перевіряти відповідність плану і стандартам проєкту |
| Збираєшся сказати "готово" | **Verification** | Виконувати фінальні перевірки перед заявою про завершення |
| Стикаєшся з незалежними завданнями | **Parallel Agents** | Запускати кілька агентів одночасно |
| Починаєш роботу над фічею | **Git Worktrees** | Ізолювати роботу в окремій гілці worktree |
| Завершуєш гілку | **Finishing a Branch** | Акуратно фіналізувати гілку і підготувати PR |

### Чому це критично важливо

**Без Superpowers** типова сесія виглядає так:

```text
Ти: Реалізуй автентифікацію користувача через OAuth

Claude: Звичайно! Давай створимо auth middleware...
        [пише 200 рядків коду]
        [розуміє, що потрібен інший підхід]
        [переписує половину]
        Готово! Я реалізував OAuth автентифікацію.
        [жодних тестів, жодного плану, жодної перевірки]
```

**З Superpowers:**

```text
Ти: Реалізуй автентифікацію користувача через OAuth

Claude: [активується Brainstorming]
        Я бачу три підходи:
        1. Passport.js — зрілий, багато стратегій, але важкий
        2. Auth.js (NextAuth) — зроблений для Next.js, простіший
        3. Кастомний з jose — легковісний, повний контроль
        Ось компроміси...

Ти: Варіант 2, беремо Auth.js

Claude: [активується Planning]
        Ось план реалізації:
        Крок 1: Встановити залежності і налаштувати провайдери
        Крок 2: Створити auth API routes
        Крок 3: Додати session middleware
        Крок 4: Захистити routes за допомогою auth guards
        ...

Ти: Виглядає добре, поїхали

Claude: [активується Execution]
        Починаємо Крок 1...
        [активується TDD] Пишемо тест для конфігурації OAuth провайдера...
        [Тест падає — чудово]
        [Пишемо реалізацію, щоб тест пройшов]
        [Тест проходить]
        Крок 1 завершено. Переходимо до Кроку 2...
        ...
        [Всі кроки виконані]
        [активується Verification]
        Запускаємо повний набір тестів... Всі проходять.
        Перевіряємо відповідність плану... Всі кроки охоплені.
        Готово.
```

Той самий запит, зовсім інший результат. План виловлює хибні кроки на ранньому етапі. Тести виловлюють регресії. Верифікація запобігає сюрпризам "у мене на машині працює".

### Workflow з плануванням запобігає головному пожирачу часу

Найбільший пожирач часу в AI-assisted кодингу — це не повільні моделі чи поганий код, а **будівництво не того, що потрібно**. Ти кажеш "додай автентифікацію", Claude інтерпретує це по-своєму, а ти хотів щось інше, і ти витратив 20 хвилин на код, який викинеш.

Workflow "план, потім виконання" від Superpowers усуває це. Claude показує тобі план *перед* написанням будь-якого коду. Ти переглядаєш, коригуєш, і тільки тоді кажеш "поїхали". 2 хвилини на перегляд плану рятують 20 хвилин, які б ти витратив на скасування неправильного підходу.

### Тригери "думай глибше" для критичних рішень

Коли тобі потрібно, щоб Claude думав особливо ретельно — архітектурні рішення, складний рефакторинг, хитрий debugging — можна підняти розмірковування до максимуму:

- **`ultrathink`** — додай це слово до будь-якого промпту для максимальних зусиль розмірковування саме для цього запиту
- **`/effort high`** — встановити постійний рівень зусиль на всю сесію
- **EXPLORE -> PLAN -> CODE -> COMMIT** — нехай Claude спочатку читає файли без запису, потім `ultrathink` на плані, критикує на крайні випадки, *потім* виконує

Це особливо потужно в поєднанні з workflow планування Superpowers. Нехай Claude brainstorm'ить на звичайних зусиллях, потім `ultrathink` на фінальному плані перед виконанням.

## 5. Serena — Навігація по коду яка розуміє твій код

Зазвичай, коли Claude потрібно зрозуміти твій код, він робить одну з двох речей: читає весь файл або запускає текстовий пошук. Обидва варіанти працюють, але обидва марнотратні. Читання 500-рядкового файлу заради однієї функції спалює контекст. Grep за іменем класу знаходить визначення, але пропускає те, що його *використовує*.

Serena змінює це. Це мовно-орієнтований MCP server, що розуміє твій код як **символи** — класи, функції, методи, змінні — а не просто текст. Він знає, що робить функція, де вона визначена і все, що її референсить.

### Яку різницю це відчуваєш

| Старий спосіб | З Serena |
|-------------|-------------|
| `grep -r "class UserService"` по всьому проєкту | `find_symbol("UserService")` — відразу туди |
| Читати весь файл, щоб зрозуміти структуру | `get_symbols_overview("user-service.ts")` — тільки контур |
| Знайти-замінити для перейменування | `rename_symbol("oldName", "newName")` — безпечно, семантично |
| Шукати вручну "хто це викликає?" | `find_referencing_symbols("UserService")` — повна відповідь |

Економія токенів сама по собі значна. Але реальний виграш — це точність: Serena знаходить *визначення*, а не просто текстові збіги.

### Налаштування для проєкту

Два кроки. Спочатку створи конфіг Serena у своєму проєкті:

```yaml
# .serena/settings.yml
project_name: "my-project"
language_server:
  language: typescript  # або python, swift, go тощо
  root_path: "src"
```

Потім скажи Claude активувати його. У своєму project CLAUDE.md:

```markdown
## Serena (Semantic Code Analysis)
At session start, call `activate_project("my-project")`.
```

(Якщо ти налаштував session start hook — описано в [Розділі 11](#11-hooks-і-захист--автоматизуй-і-захищай) — це відбувається автоматично.)

### Workflow з Serena

Найефективніший спосіб досліджувати код з Serena — патерн **масштабування**:

```text
list_dir                → Що є в цій директорії?
get_symbols_overview    → Які символи є в цьому файлі? (без тіл — тільки імена і сигнатури)
find_symbol             → Покажи повне тіло цієї конкретної функції
find_referencing_symbols → Хто це використовує? (перед тим як змінити)
```

І для редагування:

```text
replace_symbol_body   → Переписати тіло функції точно
insert_before_symbol  → Додати щось вище символу
insert_after_symbol   → Додати щось нижче символу
```

### Де Serena не допомагає

Serena індексує тільки **символи коду**. Вона не бачить:
- Конфігураційні файли (JSON, YAML, TOML)
- Markdown і документацію
- Рядкові літерали, повідомлення про помилки, коментарі
- Будь-які файли, що не є кодом

Для них потрібен текстовий пошук — про що далі...

## 6. Ripgrep — Коли текстовий пошук це те що треба

Ripgrep (`rg`) — це швидкий інструмент текстового пошуку, і Claude Code використовує його під капотом для свого інструменту `Grep`. За замовчуванням Claude використовує вбудовану версію, але можна вказати системну установку для кращої продуктивності.

### Швидке налаштування

```json
// ~/.claude/settings.json
{
  "env": {
    "USE_BUILTIN_RIPGREP": "0"
  }
}
```

```bash
# Встанови якщо ще не маєш
brew install ripgrep        # macOS
sudo apt install ripgrep    # Ubuntu/Debian
```

### Serena vs Ripgrep — розподіл ролей

Ці два інструменти ідеально доповнюють один одного, і навчити Claude коли використовувати кожен — одна з найцінніших речей, що ти можеш зробити (детальніше в [Розділі 7](#7-навчи-claude-коли-що-використовувати)):

| Бери Ripgrep коли потрібно | Бери Serena коли потрібно |
|--------------------------------|-------------------------------|
| Рядкові літерали, повідомлення про помилки | Визначення функцій/класів |
| Значення конфігів по багатьох файлах | Хто референсить цей символ? |
| Файли не-коду (docs, JSON, YAML) | Тіло конкретного методу |
| Regex патерни де завгодно | Безпечне перейменування символу |
| "Знайти кожен файл, де згадується X" | "Як виглядає API цього файлу?" |

Жоден інструмент не замінює інший. Разом вони охоплюють все.

## 7. Навчи Claude коли що використовувати

Це, мабуть, найбільш важливе вкладення у весь цей гайд, і це просто таблиця.

Claude має багато доступних інструментів — Serena, Ripgrep, Glob, Context7, Graphiti, Edit tool, Read tool. Без підказок він повертається до своїх дефолтів з тренування: читати повні файли і grep-ати текст. Це нормально, але це як використовувати молоток для всього, коли є ще відвертка і дриль.

### Таблиця

Додай це у свій глобальний CLAUDE.md. Серйозно, просто встав:

| Завдання | Кращий інструмент | Чому |
|------|-----------|-----|
| Знайти символ/клас/функцію | Serena `find_symbol` | Семантичний, мовно-орієнтований |
| Зрозуміти структуру файлу | Serena `get_symbols_overview` | Ефективно за токенами — тільки контур |
| Перевірити references перед зміною | Serena `find_referencing_symbols` | Повне — знаходить кожне використання |
| Прочитати відомий файл | `Read` | Напряму, без накладних витрат — Serena тільки коли потрібна структура |
| Шукати рядкові літерали, значення конфігів, повідомлення про помилки | `Grep` (ripgrep) | Текстові патерни, що не є символами коду |
| Шукати файли не-коду (markdown, JSON, YAML) | `Grep` (ripgrep) | Serena індексує тільки код |
| Знайти файли за іменем або патерном | `Glob` | Швидше за будь-який пошук для виявлення файлів |
| Замінити тіло функції/методу | Serena `replace_symbol_body` | Точна, символьна заміна |
| Дрібне виправлення коду (зміна на рівні рядка) | `Edit` | Швидше за Serena для хірургічних змін |
| Редагувати конфіг або файли не-коду | `Edit` | Serena не працює з не-кодом |
| Помилки типів і діагностика | TypeScript LSP / Swift LSP | Перевірка типів у реальному часі без повного білду |
| Перейменувати по всій кодовій базі | Serena `rename_symbol` | Семантичне перейменування, а не текстовий find-replace |
| Переглянути поточний API бібліотеки | Context7 | Документація в реальному часі — тренувальні дані можуть бути застарілими |
| Запустити білд, тести, лінтер | `Bash` | Команди верифікації і скрипти |
| Згадати минуле рішення або патерн | Graphiti `search_memory_facts` | Граф знань між сесіями |
| Швидкі факти про проєкт (ідентифікатори, ключі, статус) | MEMORY.md | Вже в контексті — виклик інструменту не потрібен |

**Правило:** Serena для символів коду. Grep для тексту і не-коду. Glob для файлів. Context7 для документації. Graphiti для пам'яті. LSP для перевірки типів. Read/Edit для прямої роботи з файлами.

### Чому це важливіше, ніж здається

Без цієї таблиці типова сесія Claude може:
1. Читати цілий 400-рядковий файл, щоб знайти одну функцію (40 токенів/рядок = 16 000 витрачених токенів)
2. Grep за іменем класу і отримати текстові збіги в коментарях, тестах і документації
3. Намагатись пригадати, чи змінився API Zustand з моменту дат тренування

З таблицею:
1. `get_symbols_overview` → 200 токенів для всього контуру файлу, потім читання тіла однієї функції
2. `find_symbol` → відразу до визначення
3. Context7 → отримує поточну документацію

Те саме завдання, частка токенів, краща точність. За сесію це накопичується.

## 8. Graphiti — Пам'ять що зберігається між сесіями

Ось фруструючий патерн: ти витрачаєш 30 хвилин, пояснюючи свою архітектуру Claude. Ви добре просуваєтесь. Наступна сесія — пояснюєш все знову. CLAUDE.md допомагає, але він статичний і треба пам'ятати про його оновлення.

Graphiti — інший. Це **граф знань**, куди Claude записує під час сесій і зчитує на початку. З часом він будує мережу сутностей, зв'язків і фактів про твій проєкт — з timestamps, щоб знати коли щось було вирішено і чи актуально це досі.

### Що йде в Graphiti (а що ні)

Думай про Graphiti як про місце для **рішень з контекстом** — те, що надто детальне або надто мінливе для CLAUDE.md:

**Так:**
- "Ми обрали event sourcing для orders, бо нам потрібен audit trail. Компроміс — складніші reads через projections."
- "Переписування auth middleware обумовлене юридичною відповідністю, не tech debt — рішення про scope мають віддавати перевагу відповідності над ергономікою."
- "React Query v5 вимагає огортання mutations в startTransition — виявлено під час апгрейду."

**Ні:**
- Рутинні зміни коду (для цього є git log)
- Секрети або credentials (ніколи)
- Багатослівні дампи коду (зберігай у коді, не в пам'яті)
- Все, що вже є в CLAUDE.md (не дублюй)

### Як це працює на практиці

Кожен проєкт отримує `group_id` для scoping знань. Додай це у свій project CLAUDE.md:

```markdown
## Memory (Graphiti Knowledge Graph)
When using Graphiti tools, always use `group_id="my-project"`.
```

**Початок сесії** — Claude завантажує релевантний контекст:

```text
search_memory_facts(
  query="architecture decisions patterns",
  group_ids=["my-project"]
)
```

**Кінець сесії** — Claude зберігає що дізнався:

```text
add_memory(
  group_id="my-project",
  content="Decided to use event sourcing for order processing
           because we need audit trail and temporal queries."
)
```

Найкраща частина? Обидві ці дії можна автоматизувати за допомогою hooks (дивись [Розділ 11](#11-hooks-і-захист--автоматизуй-і-захищай)), тож це просто *відбувається* без того, щоб ти про це думав.

## 9. Пам'ять — Правильна інформація в правильному місці

Claude Code має кілька систем пам'яті, і кожна робить щось різне. Використовувати не ту — як класти чек у щоденник: воно збережено, але ти ніколи не знайдеш це коли треба.

### Шпаргалка

| Що зберігаєш | Куди | Чому туди |
|--------------------|--------------|-----------|
| Постійні інструкції і правила | **CLAUDE.md** | Завжди завантажується, завжди дотримується — пишеш ти |
| Факти для швидкого доступу (ID, імена, прапорці статусів) | **MEMORY.md** | Автоматично завантажується по проєкту, легко переглянути |
| Патерни, які Claude знаходить під час роботи | **Auto Memory** | Claude записує це сам під час роботи |
| Рішення з глибоким контекстом (те *чому*) | **Graphiti** | Придатний для пошуку, з timestamps, реляційний |
| Прогрес поточного завдання | **Tasks** | Виживають після compaction контексту |
| Глибокі тематичні знання | **Skills** | Завантажуються за запитом, не завжди |

### Auto Memory — Claude навчає себе сам

Це відрізняється від MEMORY.md, що пишеш ти. Claude Code автоматично записує нотатки собі під час роботи — команди збірки, що спрацювали, інсайти з debugging, архітектуру яку виявив, виправлення стилю коду які ти давав. Вони накопичуються в `~/.claude/projects/<project>/memory/`.

Ключова різниця:
- **CLAUDE.md** = правила, що *ти* пишеш, завантажуються кожну сесію
- **Auto Memory** = патерни, що *Claude* вивчає, теж завантажуються кожну сесію
- Не дублюй між ними — якщо Claude вже дізнався щось через auto memory, тобі не потрібно додавати це в CLAUDE.md

### MEMORY.md на практиці

Знаходиться в `~/.claude/projects/<slug>/memory/MEMORY.md`, цей файл автоматично завантажується в кожній сесії для цього проєкту.

**Хороший контент для MEMORY.md:**
- "Supabase project ID: `abc123`"
- "Міграція на v2 auth: завершена 2026-01-15"
- "Назви edge functions: `process-payment`, `send-notification`"

**Поганий контент для MEMORY.md:**
- Патерни коду (замість цього читай кодову базу)
- Зведення git history (запусти `git log`)
- Рецепти виправлень (виправлення знаходиться в коді)
- Все, що вже є в CLAUDE.md

**Тримай до 200 рядків.** Для глибших тем посилайся на окремі файли:

```markdown
<!-- MEMORY.md -->
- [Supabase setup](supabase.md) — project ID, edge functions, RLS policies
- [Localization](localization.md) — supported languages, string key format
```

## 10. Context7 — Документація яка актуальна

Тренувальні дані Claude мають дату зрізу. Бібліотеки виходять оновленнями після цієї дати. Це означає, що Claude може впевнено написати код з API, який змінився два місяці тому. Context7 виправляє це, завантажуючи **живу документацію** в момент, коли Claude її потребує.

```bash
claude plugin install context7@claude-plugins-official
```

### Коли це важливо

- Написання integration коду для будь-якої бібліотеки (React, Express, Prisma, Tailwind — всіх)
- Апгрейд або міграція між версіями бібліотеки
- Debugging поведінки, специфічної для бібліотеки
- Використання бібліотеки, щодо якої Claude здається невпевненим

### Навчи Claude звертатися до неї

Додай у свій глобальний CLAUDE.md:

```markdown
## Context7
Call Context7 before writing integration code for any library.
Training data may be outdated — always verify current API.
```

### Коли пропустити

Context7 — для документації бібліотек. Не використовуй для загальних концепцій програмування, рефакторингу власного коду, бізнес-логіки або code review.

## 11. Hooks і захист — Автоматизуй і захищай

Hooks виконують дві функції: **автоматизація** (щоб ти не забував щось робити) і **захист** (щоб Claude не робив те, чого не повинен). Комбінація Claude Code hooks і git hooks створює надійну страхувальну сітку.

### Початок сесії: завантажуй все автоматично

Налаштуй hook, що запускається на початку кожної сесії:

```json
// ~/.claude/settings.json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/session-start.sh",
            "statusMessage": "Loading project context"
          }
        ]
      }
    ]
  }
}
```

Hook скрипт читає твій project CLAUDE.md, витягує Graphiti `group_id` і ім'я проєкту Serena, і вставляє нагадування:

```bash
#!/bin/bash
INPUT=$(cat)
CWD=$(echo "$INPUT" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('cwd',''))" 2>/dev/null)

GROUP_ID=""
SERENA_NAME=""

if [ -f "$CWD/CLAUDE.md" ]; then
  GROUP_ID=$(sed -n 's/.*group_id="\([^"]*\)".*/\1/p' "$CWD/CLAUDE.md" | head -1)
  SERENA_NAME=$(sed -n 's/.*activate_project("\([^"]*\)".*/\1/p' "$CWD/CLAUDE.md" | head -1)
fi

MSGS="Action required: Read ~/.claude/CLAUDE.md for global instructions.\n"
[ -n "$SERENA_NAME" ] && MSGS="${MSGS}Action required: activate_project(\"$SERENA_NAME\")\n"
[ -n "$GROUP_ID" ] && MSGS="${MSGS}Action required: search Graphiti with group_id=$GROUP_ID\n"
echo -e "$MSGS"
```

Тепер кожна сесія стартує повністю завантаженою — жодних ручних нагадувань.

### Кінець сесії: зберігай що дізнався

Stop hook запускається коли Claude збирається завершити. Використовуй його, щоб підловити незбережені знання:

```json
{
  "stopHooks": [
    {
      "type": "prompt",
      "prompt": "Before stopping, check if any architectural decisions or bug root causes were discovered. If so, save to Graphiti. Check if MEMORY.md needs new identifiers or status updates.",
      "model": "haiku"
    }
  ]
}
```

> **Чому Haiku?** Stop hook — це швидкий аудит, йому не потрібні глибокі міркування. Використання Haiku тримає його швидким і дешевим.

### Типи hooks: більше ніж просто shell скрипти

Hooks бувають трьох видів, кожен для різного рівня верифікації:

| Тип | Що робить | Найкраще для |
|------|-------------|----------|
| `command` | Запускає shell скрипт, перевіряє exit code | Linting, форматування, сповіщення |
| `prompt` | Надсилає промпт до моделі Claude для однократної оцінки | Легкий review, security checks |
| `agent` | Запускає subagent з повним доступом до інструментів (до 50 ходів) | Глибока верифікація, перевірки по багатьох файлах |

`prompt` hook ідеально підходить для перевірок "чи безпечний цей код?" без накладних витрат повного subagent. `agent` hook може читати файли, запускати пошуки і перевіряти умови по всій кодовій базі перед тим як пропустити виклик інструменту.

### Git hooks: інший рівень захисту

Не покладайся тільки на Claude Code hooks. Git hooks виловлюють проблеми в момент commit незалежно від того, як був написаний код:

- **pre-commit** — запускай linters, formatters, type checks
- **commit-msg** — enforc conventional commit формат
- **pre-push** — запускай тести перед push

Дві системи hooks доповнюють одна одну:
- **Claude Code hooks** виловлюють проблеми *під час* сесії (до того як Claude навіть записав файл)
- **Git hooks** виловлюють проблеми *в момент commit* (фінальні ворота перед тим як код потрапить в історію)

Разом вони створюють ешелоновану оборону. Claude Code hooks запобігають написанню поганих патернів. Git hooks запобігають commit-у всього, що проскочило.

### Поширені патерни hooks

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Review this file change for security issues (injection, XSS, exposed secrets). Return {\"ok\": true} if safe, {\"ok\": false, \"reason\": \"...\"} if not.",
            "model": "haiku"
          }
        ]
      }
    ]
  }
}
```

### Результат

З обома системами hooks твій workflow має кілька контрольних точок:
- **Початок сесії:** інструменти активовані, пам'ять завантажена, контекст готовий
- **Під час роботи:** security checks на редагуваннях, форматування при збереженні
- **При commit:** linting, тести, формат повідомлення
- **Кінець сесії:** рішення збережені, пам'ять оновлена, нічого не втрачено

## 12. Дозволи і безпека

Система дозволів Claude Code контролює, що Claude може робити без того, щоб питати тебе. Правильне налаштування дозволяє Claude швидко рухатися на безпечних операціях, зберігаючи захист на всьому небезпечному.

### Три рівні

```json
{
  "permissions": {
    "allow": [
      "Read(**)",                          // Читати будь-що
      "Edit(**)",                          // Редагувати будь-що
      "Bash(git:*)",                       // Всі git команди
      "Bash(npm:*)",                       // Всі npm команди
      "mcp__plugin_serena_serena__*"       // Всі Serena інструменти
    ],
    "deny": [
      "Bash(sudo *)",                      // Ніколи sudo
      "Bash(git push --force *)",          // Ніколи force push
      "Read(.env)",                        // Ніколи не читати секрети
      "Read(**/*.key)",                    // Ніколи не читати key файли
      "Read(~/.ssh/**)",                   // Ніколи не читати SSH ключі
      "Read(~/.aws/**)",                   // Ніколи не читати AWS credentials
      "Read(~/.kube/**)",                  // Ніколи не читати kube config
      "Read(~/.npmrc)",                    // Ніколи не читати npm auth
      "Read(~/.git-credentials)",          // Ніколи не читати git credentials
      "Edit(~/.bashrc)",                   // Ніколи не змінювати shell config
      "Edit(~/.zshrc)"                     // Ніколи не змінювати shell config
    ],
    "ask": [
      "Bash(curl * | bash*)"              // Питати перед pipe-to-bash
    ]
  }
}
```

### Принципи

1. **Будь щедрим з reads** — Claude працює краще, коли може вільно досліджувати
2. **Будь суворим з деструктивними операціями** — force push, schema drops, `rm -rf` на системних шляхах
3. **Завжди забороняй доступ до секретів** — `.env`, credentials, ключі, PEM файли, SSH, AWS, kube конфіги
4. **Wildcard для MCP tools** — `mcp__plugin_serena_serena__*` надає всі Serena інструменти одразу
5. **Використовуй `ask` для сірих зон** — речі, що іноді нормальні, а іноді небезпечні

### Запобігай шкідливим MCP servers у клонованих репо

Репозиторії можуть включати `.mcp.json` файли, що автоматично активують MCP servers при відкритті. Це ризик ланцюжка поставок — шкідливе репо може впровадити інструменти. Вимкни авто-активацію:

```json
{
  "enableAllProjectMcpServers": false
}
```

З цим налаштуванням, project-level MCP servers вимагають явного підтвердження.

### Bypass mode (для досвідчених користувачів)

Якщо довіряєш своєму deny list і не хочеш запитів дозволів для всього іншого:

```json
{
  "permissions": {
    "defaultMode": "bypassPermissions"
  },
  "skipDangerousModePermissionPrompt": true
}
```

Це означає, що Claude зупинятиметься і питатиме тільки коли натрапить на щось явно заборонене. Швидко, але потребує добре налаштованих deny правил. Не вмикай це поки твої deny rules не є надійними.

## 13. Subagents і команди агентів

Claude Code може запускати менші інстанції Claude для обробки підзавдань. Це одна з найпотужніших функцій як тільки виходиш за межі дефолтів.

### Модель оркестрування

Найефективніший спосіб використовувати Claude Code — уявити головну сесію як **оркестратор**: вона планує, делегує, перевіряє і рецензує. Реальна робота з реалізації делегується subagents.

Це не просто зручна ментальна модель — саме так структурують найкращі конфігурації:

| Роль | Модель | Коли використовувати |
|------|-------|----------|
| **Оркестратор** (головна сесія) | `opus` | Завжди — планування, review, валідація, координація |
| **Агент реалізації** | `sonnet` | Написання коду, тестів, стандартні рефакторинги |
| **Агент дослідження** | `haiku` | Широкий пошук по кодовій базі, швидкі запити, пошук файлів |
| **Критичний subagent** | `opus` | Коли складність або вплив вимагають зробити все правильно |

Додай у свій глобальний CLAUDE.md:

```markdown
## Orchestration Model
Main session = orchestrator (plans, delegates, validates, reviews)
- haiku — exploration, quick lookups
- sonnet — implementation, tests, code review, standard QA
- opus — architecture, security-critical, complex refactors
```

### Правила subagents, що окупаються

Ці правила запобігають найпоширенішим збоям subagents:

- **Паралельно за замовчуванням.** Для завдань, що зачіпають 5+ незалежних файлів, запускай паралельні subagents (по 5-8 файлів кожен). Послідовна обробка великих завдань гарантовано призводить до деградації контексту в оркестраторі.
- **Знання інструментів.** Subagents дотримуються тих самих правил вибору інструментів, що й головна сесія — Serena для коду, Grep для тексту тощо. Не давай їм за замовчуванням читати цілі файли.
- **Самоперевірка.** Subagents повинні збирати, перевіряти типи, запускати лінтер і виконувати відповідні тести перед тим, як повідомляти про завершення. "Компілюється" — це не доказ верифікації.
- **Оркестратор перевіряє двічі.** Головна сесія незалежно переглядає результати subagents перед прийняттям. Довіряй, але перевіряй.

### Визначення власних subagent

Крім вбудованих агентів, ти можеш визначати власних спеціалізованих агентів в `.claude/agents/`. Вони багаторазові між сесіями і можуть накопичувати знання з часом.

Приклад — code reviewer, що знає твій проєкт:

```markdown
<!-- .claude/agents/code-reviewer.md -->
---
name: code-reviewer
description: Reviews code changes for bugs, security issues, and adherence to project conventions
model: sonnet
memory: project
tools:
  - Read
  - Grep
  - Glob
---

You are a code reviewer for this project. Review changes for:
- Logic errors and edge cases
- Security vulnerabilities
- Adherence to project conventions in CLAUDE.md
- Test coverage gaps

Be specific. Cite line numbers. Suggest fixes, don't just flag problems.
```

Налаштування `memory: project` означає, що цей агент накопичує знання про кодову базу в `~/.claude/agent-memory/code-reviewer/` між сесіями. Чим більше використовуєш, тим краще він розуміє твій проєкт.

Визначення агентів під твої конкретні потреби — це game-changer: security reviewer, test writer, архітектурний консультант. Кожен може мати свою модель, інструменти і пам'ять.

### Команди агентів (Agent Teams)

> *Я ще не використовував Agent Teams, але варто знати про них для складних проєктів.*

Agent Teams виходять за межі простих subagents. Замість одностороннього делегування батько-дитина, команди мають:
- **Спільні списки завдань** з відстеженням залежностей
- **Спілкування між рівноправними агентами**
- **Паралельне виконання** — кожен учасник команди є повноцінним інстансом Claude Code

Думай про це як перехід від "один розробник з асистентами" до "невелика команда, що працює разом."

Увімкни це:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

Найкращі випадки використання: паралельний code review (security + performance + test reviewers), розробка незалежних модулів, cross-layer зміни де фронтенд і бекенд повинні рухатися разом.

**Рекомендації по розміру:** 3-5 учасників, 5-6 завдань на учасника. Кожен учасник спалює свої токени, тому це потужний інструмент — використовуй його коли паралелізм виправдовує вартість.

## 14. Управління контекстним вікном

Це, мабуть, найважливіший операційний висновок у цьому гайді: **Claude працює найкраще коли контекстне вікно менше 200K токенів.** Не 500K, не 1M — менше 200K. Так, вікно *підтримує* 1M, але якість погіршується в міру заповнення. Думай про це як про RAM: у твого комп'ютера є 64 GB, але якщо ти використовуєш 60 GB — все гальмує.

З цього випливає пряме практичне наслідування: **compact і продовжити майже завжди краще ніж починати нову сесію** для схожої роботи. Коли ти робиш `/compact`, Claude зберігає ключові рішення та контекст, відкидаючи шум. Сесія після compact з 50K сфокусованого контексту перевершує нову сесію, де Claude має заново перечитувати все з нуля — і значно перевершує роздуту сесію на 500K, де важливі інструкції поховані в старих виводах інструментів.

### Ключові команди

| Команда | Що робить | Коли використовувати |
|---------|-------------|-------------|
| `/compact` | Стискає историю розмови, зберігаючи ключовий контекст | Коли сесія здається мляво або переключаєшся між підзавданнями |
| `/compact Focus on the API changes` | Направлене стиснення — каже Claude що зберегти | Коли хочеш щоб конкретний контекст вижив |
| `/clear` | Стирає историю розмови повністю | Між непов'язаними завданнями — найменш використовувана команда |
| `/btw` | Задати побічне питання, що не потрапляє в историю | Швидкі запити, що не повинні розростати контекст |
| `/context` | Показує розбивку токенів по категоріях | Діагностика того, що з'їдає твій контекст |
| `Ctrl+B` | Відправити запущений subagent у фоновий режим | Коли subagent довго виконується і ти хочеш зробити щось інше |

### Compact vs clear vs нова сесія

Це рішення, яке ти приймаєш десятки разів на день. Ось правило:

| Ситуація | Найкраща дія | Чому |
|-----------|------------|-----|
| Переключаєшся на **непов'язану** роботу | `/clear` | Старий контекст — чистий шум для нового завдання |
| Продовжуєш **схожу** роботу після завершення шматка | `/compact` | Зберігай рішення, відкидай вивід інструментів |
| Сесія гальмує або Claude забуває інструкції | `/compact` з підказкою | Скидай шум, зберігаючи важливе |
| Починаєш повністю **інший проєкт** | Нова сесія | Інший CLAUDE.md, інший Serena проєкт, все інше |

Ключовий інсайт: **`/compact` + продовжити перемагає нову сесію**, коли робота пов'язана. Стиснута сесія зберігає твої архітектурні рішення, узгоджені патерни та прогрес завдань. Нова сесія стартує холодною і має заново все це відкривати.

### Практичні звички

**Використовуй `/clear` агресивно між непов'язаними завданнями.** Закінчив debugging API endpoint? `/clear` перед початком роботи на фронтенді. Вартість повторного читання кількох файлів — ніщо в порівнянні з тим, щоб тягнути 100K нерелевантного debugging контексту.

**Compact завчасно, не реактивно.** Не чекай поки Claude почне плутатися. Після завершення великого кроку — `/compact`, щоб тримати контекст легким. Думай про це як прибирання столу між завданнями — невелика звичка, що дає складний ефект.

**Кастомні інструкції `/compact` працюють.** Замість просто `/compact`, спробуй:

```text
/compact Keep the architectural decisions and test commands,
         summarize the debugging session
```

Це направляє те, що виживе після стиснення. Корисно коли збираєшся переключити фокус, але хочеш зберегти конкретні рішення в контексті.

**`/btw` для швидких запитів.** Потрібно перевірити сигнатуру функції або значення конфігу, але не хочеш захаращувати розмову? `/btw what's the return type of getUserById?` — відповідь з'являється в overlay, що можна закрити, ніколи не потрапляє в историю.

**Subagents — найкращий захист контексту.** Делегуй дослідницьку роботу subagents. Вони працюють у власному контексті, повертають тільки відповідь і тримають твою основну сесію чистою. "Використай subagent щоб дослідити збої тестів в auth модулі" — краще ніж досліджувати напряму і заповнювати сесію 50 читаннями файлів.

### Гігієна контексту — захисні звички

Це тихі вбивці довгих сесій. Жодна з них не дає очевидних помилок — речі просто поступово погіршуються:

- **Перечитуй перед редагуванням після значного обсягу роботи.** Авто-compaction відкидає більш ранні читання файлів. Якщо ти зробив 20 викликів інструментів з моменту останнього читання файлу, Claude може редагувати на основі застарілого контексту. Рішення просте: перечитай, потім редагуй.
- **Великі файли потребують читання частинами.** Кожен виклик `Read` повертає максимум 2 000 рядків. Для файлів, що наближаються до цього, використовуй параметри `offset` і `limit`. Ніколи не вважай, що одне читання захопило все.
- **Стеж за скороченими результатами інструментів.** Великі результати пошуку мовчки скорочуються. Якщо пошук повертає підозріло мало результатів — перезапусти з вужчим охопленням: конкретна директорія, суворіший glob-патерн.
- **Обмеження файлу в 500 рядків.** Файли понад 500 рядків важко ефективно опрацьовувати Claude. Якщо файл виростає за цю межу — розбий його: один клас або модуль на файл — хороший стандарт за замовчуванням.

### Що виживає після compaction

- CLAUDE.md: перечитується з диску при кожному compaction — завжди виживає
- Auto Memory / MEMORY.md: завжди завантажується — завжди виживає
- Tasks: виживають після compaction — ось чому вони важливі
- Інструкції в розмові: **НЕ виживають** якщо не записані в CLAUDE.md

Якщо інструкція, яку ти давав Claude, зникла після compaction — вона була тільки в розмові. Запиши її в CLAUDE.md або skill якщо вона має бути постійною.

## 15. Git Worktrees — Паралельні сесії

Claude Code *створений* для паралелізму. Це не запізнілий додаток — це базовий архітектурний принцип. Subagents працюють паралельно всередині сесії. Кілька сесій працюють паралельно в різних worktrees. І ти можеш поєднувати обидва: головна сесія оркеструє subagents, поки інші сесії незалежно працюють у своїх worktrees.

Коли ти засвоюєш це, перестаєш думати про Claude як "один асистент робить одну річ" і починаєш думати про нього як "команда, яку я можу розгортати по всьому кодовій базі."

### Ідея

Git worktree — це окремий checkout того ж репо в іншій директорії, на власній гілці. Кожна сесія Claude працює у власному worktree — різні файли, різна гілка, жодних конфліктів.

### Вбудована підтримка

Claude Code має першокласну підтримку worktrees, а plugin Superpowers включає skill **Using Git Worktrees**, що керує workflow:

```bash
# Запустити сесію в новому worktree
claude --worktree feature-auth
```

Це створює `.claude/worktrees/feature-auth/` з виділеною гілкою.

### Патерн для досвідчених користувачів

Команди з високою продуктивністю запускають 5-15 сесій Claude одночасно — деякі у вкладках терміналу, деякі у веб-застосунку — кожна у своєму worktree. Фронтенд в одному, бекенд в іншому, тести в третьому. Всі працюють паралельно без конфліктів, і кожна сесія відкриває власний PR по завершенні.

### Два рівні паралелізму

Думай про це як про дві взаємодоповнюючі стратегії:

**Всередині сесії — subagents.** Claude відправляє субагенти Haiku/Sonnet/Opus для паралельного виконання підзавдань. Досліджуй три підходи одночасно. Запускай code review та тести водночас. Вивчай кодову базу в одному subagent, поки плануєш в іншому. Це тримає контекст головної сесії чистим і прискорює роботу.

**Між сесіями — worktrees.** Кілька повноцінних сесій Claude, кожна у своєму worktree, працюють над незалежними завданнями. Саме так ти паралелізуєш на рівні фіч — автентифікація в одній сесії, API endpoints в іншій, міграції бази даних у третій.

Поєднання — ось де справжня сила. Кожна worktree-сесія може *сама* відправляти subagents. В результаті виходить дерево паралельної роботи, яку людська команда координувала б днями.

### Поради по налаштуванню

**Додай директорію worktrees в gitignore:**

```gitignore
.claude/worktrees/
```

**Копіюй env файли в нові worktrees** за допомогою `.worktreeinclude`:

```text
# .worktreeinclude — gitignored файли для копіювання в кожен новий worktree
.env.local
.env.development
```

Копіюються тільки gitignored файли — tracked файли приходять з git checkout автоматично.

### Worktrees для subagents

Ти також можеш давати subagents власні worktrees для справді ізольованої паралельної роботи:

```markdown
<!-- .claude/agents/my-agent.md -->
---
isolation: worktree
---
```

Worktree автоматично очищається якщо агент не робить змін.

## 16. Git і code review

### Conventional commits

Визнач свій формат один раз у глобальному CLAUDE.md і ніколи більше про це не думай:

```markdown
## Git
Commits: <type>(<scope>): <description> (50 char max subject)
Body: blank line + bullet list, 72 char wrap
Types: feat fix docs style refactor perf test build ci chore revert
Rules: Never commit to main. Never force push. Never skip hooks.
git commit is auto-approved. git push requires confirmation.
```

### Code review: використовуй обидва підходи

Найкраще налаштування code review використовує **два взаємодоповнюючі підходи**:

**1. [CodeRabbit](https://www.coderabbit.ai/) — хмарний AI review для PR**

CodeRabbit автоматично переглядає твої pull requests на GitHub. Він виловлює проблеми зі свіжого погляду — він не бачив контексту твоєї сесії, тому знаходить те, що ти і Claude обидва пропустили.

```bash
claude plugin install coderabbit@claude-plugins-official
```

```text
/coderabbit:review    — Запустити CodeRabbit review на поточних змінах
```

**Workflow для CodeRabbit reviews:**
1. Вирішити всі коментарі — включаючи пропозиції "nice to have"
2. Розрізняти кожен review thread після виправлення
3. Написати `@coderabbitai resolve` для очищення незакритих коментарів
4. Написати `@coderabbitai full review` для нового проходу
5. CodeRabbit перемикається на APPROVED коли всі threads чисті

**2. Локальні subagent reviews — негайний, контекстний feedback**

Claude Code може запускати review subagents, що перевіряють твою роботу відповідно до плану і стандартів проєкту прямо у твоїй сесії. Plugin Superpowers робить це автоматично після завершення основних кроків реалізації.

Ти також можеш запустити це вручну:

```text
/code-review
```

Або визнач кастомного review агента (дивись [Розділ 13](#13-subagents-і-команди-агентів)), що накопичує знання про твою кодову базу з часом.

**Навіщо обидва?** Локальні reviews швидкі і контекстні — вони знають план і відразу виловлюють відхилення. CodeRabbit reviews свіжі і незалежні — вони виловлюють проблеми, які session bias міг пропустити. Разом вони дають ешелоновану оборону.

### Верифікація: покроково, а не тільки в кінці

Не чекай PR для перевірки. Після кожного логічного кроку:

1. **Білд / перевірка типів** (наприклад, `tsc --noEmit`, `xcodebuild`)
2. **Лінтер** (наприклад, `eslint . --quiet`, `swiftlint`)
3. **Запуск відповідних юніт-тестів**

Це стосується і subagents — вони повинні перевірятись перед тим, як повідомляти про завершення (дивись [Розділ 13](#13-subagents-і-команди-агентів)). Перед створенням PR запусти повний набір тестів. Додай команди верифікації свого проєкту в його CLAUDE.md, щоб Claude завжди знав, як перевіряти свою роботу.

## 17. Управління сесіями

Сесії в Claude Code не одноразові — їх можна відновлювати, називати і перемикатися між ними. Це важливо для багатоденної роботи і коли потрібно перемикати контекст.

### Ключові команди

| Команда | Що робить |
|---------|-------------|
| `claude --continue` | Відновити найбільш останню сесію |
| `claude --resume` | Відкрити picker для вибору яку відновити |
| `claude -n auth-refactor` | Запустити іменовану сесію (легше знайти потім) |
| `/rename auth-refactor` | Перейменувати поточну сесію |
| `claude --from-pr 123` | Відновити сесію, пов'язану з конкретним PR |

### Shortcut'и для session picker

У `/resume` picker:
- **`P`** — переглянути сесію
- **`R`** — перейменувати
- **`B`** — фільтрувати по поточній git гілці
- **`A`** — перемикати між поточною директорією і всіма проєктами

### Префікс `!`

Потрібно виконати швидку shell команду без зайвого round-trip до інструменту? Додай префікс `!`:

```text
! git status
! npm test
! ls -la src/
```

Вихід потрапляє прямо в контекст розмови. Швидше ніж просити Claude запустити, і іноді це все що потрібно.

## 18. Status line — Бачи що відбувається

Claude Code підтримує кастомний status line в нижній частині терміналу, що показує інформацію про твою сесію в реальному часі. Налаштуй за допомогою shell скрипту в `~/.claude/statusline.sh`.

Хороший status line показує:
- **Використання контексту** — наскільки заповнене контекстне вікно (зелений < 50%, жовтий 50-79%, червоний 80%+)
- **Вартість сесії** — скільки ти витратив в цій сесії
- **Час, що минув** — як довго ти вже працюєш
- **Prompt cache hit rate** — наскільки ефективно повторно використовується контекст

Це дає тобі моментальне усвідомлення коли робити `/compact` або `/clear`, і допомагає виробити інтуїцію про те, що дорого, а що дешево.

## 19. Трекінг завдань — Не загубись

Коли твоя розмова стає довгою, Claude Code стискає старіші повідомлення щоб залишитись в межах контекстних обмежень. Зазвичай це нормально — але це означає, що Claude може забути де він знаходився в багатокроковому плані.

Tasks виживають після цього стиснення. Вони завжди видимі, навіть після compaction.

```text
TaskCreate   — Створити завдання
TaskUpdate   — Позначити як in_progress, completed або blocked
TaskList     — Побачити всі поточні завдання
```

**Практика:** Для будь-якої роботи з більш ніж 2-3 кроками, створюй завдання наперед. Позначай кожне `in_progress` коли починаєш, `completed` коли закінчуєш — одразу, а не пакетно в кінці.

> Tasks — для поточної сесії. Для інформації, що повинна зберігатися між сесіями, використовуй MEMORY.md або Graphiti.

## 20. Типові помилки — Що варто уникати

Це помилки, що найбільше марнують час з Claude Code. Кожна з них легко допускається і легко виправляється, як тільки знаєш що шукати.

### Сесія "всього і відразу"

**Патерн:** Ти дебажиш API endpoint, потім виправляєш CSS баг, потім пишеш міграцію, потім рефакториш тест — все в одній сесії. До кінця Claude тягне 500K змішаного контексту і його відповіді стають розмитими.

**Виправлення:** `/clear` між непов'язаними завданнями. Це здається марнотратним ("але я тільки що мав весь цей контекст!"), але свіжий старт з правильним контекстом щоразу б'є роздуту сесію.

### Петля виправлень

**Патерн:** Claude робить щось неправильно. Ти виправляєш. Він робить неправильно знову, трохи по-іншому. Ти виправляєш знову. Через три раунди ти розчарований, а Claude заплутаний — кожне виправлення додавало суперечливого контексту.

**Виправлення:** Після двох невдалих виправлень однієї і тієї ж проблеми — зупинись. `/clear` і напиши кращий початковий промпт. Проблема не в розумінні Claude — вихідна інструкція була недостатньо чіткою, і кожне виправлення робить контекст шумнішим.

### Переспецифікований CLAUDE.md

**Патерн:** Твій CLAUDE.md — 800 рядків детальних правил. Claude ігнорує половину з них, бо важливі правила заховані в шумі. Ти додаєш більше правил щоб компенсувати. Стає гірше.

**Виправлення:** Кожен рядок в CLAUDE.md повинен пройти перевірку: *чи призведе видалення цього до конкретної помилки Claude?* Якщо Claude вже робить правильно без правила — видали правило. Цільуйся на ~100-200 рядків. Перемісти глибокі знання в skills, що завантажуються за запитом.

### Розрив "довіряй, але не перевіряй"

**Патерн:** Claude пише правдоподібно виглядаючий код. Ти швидко переглядаєш, кажеш "виглядає добре" і рухаєшся далі. Пізніше виявляється, що він не обробляє крайні випадки, або непомітно зламав щось інше.

**Виправлення:** Завжди надавай критерії верифікації — тести для запуску, скрипти для виконання, конкретну поведінку для перевірки. Verification skill Superpowers робить це автоматично, але ти також можеш просто сказати: "Перед тим як завершимо, запусти набір тестів і покажи мені вихід."

### Нескінченне дослідження

**Патерн:** Ти просиш Claude "дослідити проблеми з продуктивністю." Claude читає 80 файлів, запускає 30 пошуків і заповнює твій контекст вичерпним аналізом, який тобі не потрібен.

**Виправлення:** Обмежуй свої запити. "Перевір чи N+1 запит у getUserOrders спричиняє сповільнення" замість "досліди продуктивність." Або делегуй subagent: "Використай subagent щоб дослідити проблеми з продуктивністю в модулі orders" — дослідження відбувається в окремому контексті і повертає тільки результати.

### Брехня "я додам тести пізніше"

**Патерн:** Claude реалізує фічу без тестів. Ти плануєш додати їх пізніше. Ти ніколи не робиш. Наступна зміна тихо ламає фічу.

**Виправлення:** TDD workflow Superpowers обробляє це автоматично — тести йдуть спочатку, а не після. Якщо не використовуєш Superpowers, зроби це правилом у своєму CLAUDE.md: "Write failing tests before implementation. No exceptions."

## 21. Синхронізація конфігу — Однакове налаштування скрізь

Якщо ти розробляєш на більш ніж одній машині — скажімо, робочий ноутбук і особистий — ти хочеш щоб твоє налаштування Claude Code було однаковим на обох. Той самий CLAUDE.md, ті самі hooks, ті самі permissions, ті самі plugins.

### cc-config-sync

[cc-config-sync](https://www.npmjs.com/package/cc-config-sync) — це CLI інструмент, що зберігає твій конфіг Claude Code в git репо і push/pull між машинами:

```bash
npm install -g cc-config-sync

cc-config-sync pull        # Захопити конфіг поточної машини в репо
cc-config-sync push --yes  # Застосувати конфіг репо до поточної машини
cc-config-sync status      # Перевірити що відрізняється
cc-config-sync push --dry-run  # Переглянути без запису
```

### Що синхронізується

- `~/.claude/CLAUDE.md` (глобальні інструкції)
- `~/.claude/settings.json` (permissions, hooks, plugins)
- `~/.claude/hooks/*.sh` (hook скрипти)
- `~/.claude/plugins/installed_plugins.json`
- CLAUDE.md, settings і memory файли по-проєктно

### Одна річ, за якою варто стежити

`settings.local.json` містить машино-специфічні перевизначення — наприклад, шляхи що відрізняються між машинами. Переглядай їх перед push між середовищами. Те, що працює на одній файловій системі, може не підходити для іншої.

## З чого починати

Якщо налаштовуєш з нуля, ось порядок, що дасть найбільшу цінність найшвидше:

1. **Встанови основні plugins** — superpowers, claude-md-management, skill-creator, context7, serena, code-review, coderabbit
2. **Згенеруй перший CLAUDE.md** — використай `/init` для скаффолдингу, потім вдосконали з `/claude-md-improver`
3. **Додай таблицю вибору інструментів** у свій глобальний CLAUDE.md — ця одна таблиця змінює те, як Claude працює
4. **Налаштуй permissions** — дозволяй reads вільно, забороняй небезпечне і секрети
5. **Налаштуй hooks** — session start для автоматичної активації інструментів, session end для збереження в пам'ять
6. **Напиши свій перший project CLAUDE.md** — архітектура, конвенції, ключові файли, команди збірки
7. **Налаштуй Serena** для свого основного проєкту — економія токенів починається одразу
8. **Налаштуй Graphiti** для постійної пам'яті — рішення перестають губитися між сесіями
9. **Створи свій перший skill** — візьми щось, що ти постійно пояснюєш, і зроби це за запитом
10. **Довірся Superpowers** — дозволь йому керувати workflow планування, TDD і верифікації

Кожен крок будується на попередніх. Не обов'язково робити все одразу — навіть кроки 1-5 дадуть помітну різницю.

# Claude Code: лучшие практики

[English](README.md) | **Русский** | [Українська](README.uk.md) | [Español](README.es.md)

> **Для кого это руководство?** Для разработчиков, которые уже используют Claude Code и хотят перейти от «ну, работает» к «реально в 10 раз продуктивнее». Независимо от того, пользуетесь вы им неделю или полгода — здесь найдёте что-то полезное.

Большинство людей устанавливают Claude Code, набирают несколько промптов и получают приличные результаты. Но между случайным использованием и настроенной системой — пропасть. Настроенной так, что Claude по-настоящему *понимает* ваши проекты, помнит ваши решения и работает так, как работаете вы. Это руководство закрывает тот пробел.

Начнём с фундамента — плагины, MCP-серверы и команды — а затем перейдём к практикам, которые дают эффект накопления: как структурировать инструкции, когда какой инструмент использовать и как сделать так, чтобы каждая новая сессия не начиналась с нуля.

## 1. Экосистема расширений

Claude Code из коробки — хорош. Claude Code с правильными расширениями — совершенно другой опыт.

Есть три способа расширить его возможности, и они работают вместе:

### Плагины — главное

Плагины объединяют всё: slash-команды, специализированных агентов, hooks и иногда MCP-инструменты. Думайте о них как о «пакетах функций» для Claude Code.

```bash
# Открыть маркетплейс плагинов
/plugins

# Или установить напрямую
claude plugin install superpowers@claude-plugins-official
```

Вот плагины, которые стоит поставить сразу — дальше мы подробно разберём самые важные:

| Плагин | Зачем нужен |
|--------|------------|
| **superpowers** | Планирование, брейнсторминг, TDD, отладка, worktrees, code review — структурированные воркфлоу, которые делают Claude дисциплинированным |
| **claude-md-management** | Написание и аудит CLAUDE.md с помощью ИИ |
| **skill-creator** | Превращает любые знания о проекте в переиспользуемый skill |
| **serena** | Семантическая навигация по коду — находит символы, ссылки, редактирует по смыслу |
| **context7** | Живая документация по библиотекам — Claude не будет галлюцинировать устаревшее API |
| **code-review** | Воркфлоу для code review PR с параллельными subagent-рецензентами |
| **coderabbit** | Интеграция с [CodeRabbit](https://www.coderabbit.ai/) AI review |
| **feature-dev** | Разработка фич с исследованием кодовой базы |
| **playwright** | Автоматизация браузера и тестирование |
| **plugin-dev** | Инструменты для создания собственных плагинов |

### MCP-серверы — внешние инструменты

MCP (Model Context Protocol) серверы подключают Claude к внешним системам — вашей IDE, трекеру задач, графу знаний, инструментам мониторинга. Они предоставляют сырые инструменты, которые Claude вызывает напрямую.

```bash
claude mcp add <name> -- <command>
claude mcp list
```

Часто используемые:

| Сервер | Что делает |
|--------|-----------|
| **Graphiti** | Граф знаний для постоянной памяти |
| **JetBrains** | Сборка, тесты и рефакторинг прямо из IDE |
| **Atlassian** | Jira + Confluence (часто предустановлен через OAuth) |
| **Sentry** | Отслеживание ошибок и расследование инцидентов |
| **Docker MCP** | Каталог MCP-серверов, которые можно установить по запросу |

### Slash-команды — интерфейс пользователя

Slash-команды — это то, как вы вызываете skills из плагинов. Наберите `/` и увидите всё доступное:

```text
/commit              Create a git commit
/code-review         Review a pull request
/revise-claude-md    Update CLAUDE.md with session learnings
/skill-creator       Create a new skill from knowledge
/simplify            Review and simplify changed code
```

Это был быстрый обзор. Теперь перейдём к тому, что реально меняет ситуацию.

## 2. CLAUDE.md — инструкции, которые работают

Если вы сделаете из этого руководства только одно — **напишите хороший CLAUDE.md.**

Каждый раз при старте сессии Claude читает ваши CLAUDE.md-файлы. Это ваши постоянные инструкции — то, что иначе пришлось бы повторять в каждом разговоре. Правильно разделите глобальный и проектный файл, и Claude начнёт каждую сессию уже зная, как вы работаете.

### Два файла, две задачи

| | Глобальный (`~/.claude/CLAUDE.md`) | Проектный (`<repo>/CLAUDE.md`) |
|-|-------------------------------------|-------------------------------|
| **Загружается** | В каждой сессии, в каждом проекте | Только в этом проекте |
| **Думайте о нём как о** | Вашем профиле разработчика | Справочнике по проекту |
| **Содержит** | Как работаете *вы* | Как работает *эта кодовая база* |

### Глобальный файл: профиль разработчика

Это рассказывает Claude о вас — для всех проектов. Напишите один раз, улучшайте постепенно:

- **Какие инструменты доступны** и когда каждый из них активировать
- **Как вы оформляете commits** и какая у вас стратегия веток
- **Ваша планка качества кода** — DRY, SOLID, строгая типизация — всё, что важно
- **Насколько тщательно проверять работу** перед тем, как сказать «готово»
- **Какая ИИ-модель для какой задачи** — Haiku для поиска, Opus для архитектуры
- **Ваши активные проекты** — названия, пути, Jira-ключи для быстрого доступа

Вот как выглядит хороший скелет глобального CLAUDE.md:

```markdown
# Global Claude Code Instructions

## Tools & MCP Servers
| Tool | Purpose | When to Activate |
|------|---------|-----------------|
| Serena | Code navigation | activate_project("<name>") at session start |
| Graphiti | Persistent memory | search_memory_facts at session start |
| Context7 | Library docs | Before writing integration code |

## Git
Commits: <type>(<scope>): <description>
Never commit to main. Never force push. PRs require confirmation.

## Code Quality
Read before modifying. Comments explain *why*. No `any` types.

## Subagents
- haiku — exploration, quick lookups
- sonnet — implementation, tests, code review
- opus — architecture, security-critical decisions
```

> **Совет профи:** Стремитесь к 100-200 строкам. Борис Черни (создатель Claude Code) говорил, что «около 100 строк работает лучше, чем 800». Каждая строка должна проходить проверку: *если убрать это правило, Claude будет делать ошибки?* Если Claude и так делает правильно без правила — удалите правило.

### Проектный файл: справочник по кодовой базе

Здесь живут специфические знания о проекте. Думайте об этом как о документе для онбординга нового члена команды:

- **Архитектура** — модели данных, ключевые файлы, как всё связано
- **Команды сборки/тестов/линтинга** — точные команды, без угадывания
- **Стандарты кода, специфичные для проекта** — «используй `Logger`, не `print()`»
- **Типичные ловушки** — то, что стреляет именно в *этой* кодовой базе
- **Указатели на интеграции** — Jira-ключ, Graphiti `group_id`, имя проекта в Serena

> **Что сюда не нужно:** Общие советы по коду (это в глобальный файл), инструкции по активации инструментов (тоже глобальный), всё временное (это задачи или память).

### Начинаете с нуля? Используйте `/init`

Если вы настраиваете CLAUDE.md для существующего проекта — не стартуйте с пустого файла. Команда `/init` в Claude Code исследует кодовую базу — package-файлы, конфиги, структуру директорий, существующую документацию — и автоматически генерирует стартовый CLAUDE.md:

```bash
# Интерактивный многоэтапный процесс: исследование, потом написание
CLAUDE_CODE_NEW_INIT=1 claude /init
```

Хорошая отправная точка. Затем улучшайте постепенно с помощью `/revise-claude-md` (об этом ниже).

### Path-specific правила: правильный контекст для нужных файлов

> *Я сам ещё не использовал это, но слишком полезная штука, чтобы не упомянуть.*

Вместо того чтобы загружать все правила всё время, можно привязать правила к определённым паттернам файлов через `.claude/rules/`:

```yaml
# .claude/rules/api-conventions.md
---
paths:
  - "src/api/**/*.ts"
---

## API Design Rules
- All endpoints return { data, error, meta }
- Use Zod for request validation
- Rate limit: 100 req/min per user
```

Этот файл загружается только когда Claude работает с файлами, соответствующими `src/api/**/*.ts`. Конвенции тестов загружаются только при редактировании тестов. Паттерны фронтенда — только при работе с компонентами. Та же идея, что и skills (Раздел 3), но триггером служит путь к файлу, а не задача.

### Не пишите в одиночку

Плагин **claude-md-management** предоставляет два skill'а, которые делают всю тяжёлую работу:

```bash
claude plugin install claude-md-management@claude-plugins-official
```

| Команда | Что делает |
|---------|-----------|
| `/claude-md-improver` | Аудирует ваш CLAUDE.md на пробелы, несоответствия и пропущенные лучшие практики |
| `/revise-claude-md` | После рабочей сессии фиксирует, что вы узнали — новые паттерны, ловушки, исправления |

**Привычка, которая окупается:** Запускайте `/revise-claude-md` в конце каждой сессии, где вы что-то открыли для себя в проекте. Со временем ваш CLAUDE.md превращается из заглушки в исчерпывающий, проверенный в бою справочник — без необходимости писать его с нуля.

### Наследование CLAUDE.md (монорепо и вложенные директории)

Claude Code идёт вверх по дереву директорий и загружает каждый найденный CLAUDE.md:
- `~/.claude/CLAUDE.md` — ваш глобальный профиль
- `./CLAUDE.md` или `./.claude/CLAUDE.md` — корень проекта (коммитьте в git для команды)
- CLAUDE.md-файлы в дочерних директориях — загружаются лениво, когда Claude работает в этих директориях

В монорепо это значит, что корневой CLAUDE.md покрывает общие конвенции, а каждая директория пакета может иметь свои переопределения. Используйте `claudeMdExcludes` в settings, если файлы других команд создают шум.

## 3. Skills — знания по требованию

Вот проблема, с которой вы быстро столкнётесь: ваш CLAUDE.md растёт. Вы добавляете правила — API-конвенции, паттерны тестирования, стандарты БД, чеклисты деплоя — и в итоге каждая сессия загружает 2000 токенов контекста, нужен он или нет.

Skills решают это. Это как секции CLAUDE.md, которые загружаются **по требованию**, а не всегда.

### Суть за 30 секунд

Skill — это markdown-файл с описанием. Когда Claude начинает задачу, он проверяет: «Есть ли skill, подходящий для того, что я собираюсь делать?» Если да — загружает его. Если нет — не мешает.

```text
Вы: Fix the flaky test in UserService

Claude: [видит, что это задача по тестированию]
        [загружает skill test-driven-development]
        [загружает skill testing-conventions вашего проекта]
        [работает с полным контекстом тестирования — ничего лишнего]
```

### До и после

**До** — всё запихнуто в CLAUDE.md:

```markdown
## API Design Rules
- All endpoints return { data, error, meta }
- Use Zod for request validation
- Rate limit: 100 req/min per user
- Error codes follow RFC 7807
... (ещё 50 строк)

## Database Conventions
- All tables have created_at, updated_at
- Use soft deletes
- Indexes on all foreign keys
... (ещё 40 строк)
```

Всё это загружается в каждой сессии. Исправляете CSS-баг? Вы всё равно тащите в контекст правила по API и базе данных.

**После** — лаконичный CLAUDE.md, знания по требованию:

```markdown
## Available Skills
- `/api-design` — API conventions, response format, error codes
- `/db-conventions` — Database patterns, migrations, indexes
```

Теперь API-контекст загружается только когда вы реально работаете с эндпоинтами. Те же знания, меньше расхода, лучше фокус.

### Как создать skill

Не нужно писать skill-файлы вручную. Плагин **skill-creator** сделает всё за вас:

```bash
claude plugin install skill-creator@claude-plugins-official
```

Просто опишите, что вы знаете:

```text
Вы: /skill-creator

Claude: Какие знания вы хотите превратить в skill?

Вы: Наше API всегда возвращает ответы в таком формате: { data, error, meta }.
     Для всей валидации запросов используем Zod. Rate limit — 100 req/min.
     Ошибки по RFC 7807. Все эндпоинты версионированы под /api/v1/...

Claude: [Создаёт хорошо структурированный skill-файл с правильным frontmatter,
         условиями срабатывания и организованными инструкциями]
```

Вы даёте знания, плагин занимается форматированием и описанием триггеров.

### Что делает skill-файл рабочим

```markdown
---
name: api-design
description: Use when creating or modifying API endpoints — covers response
  format, validation, error handling, and versioning conventions
---

## Response Format
All endpoints return:
{ "data": ..., "error": null | { ... }, "meta": { ... } }

## Validation
Use Zod schemas for all request bodies...
```

Поле `description` делает основную работу. Оно говорит Claude, *когда* загружать skill — пишите его как условие триггера: «Use when [делаешь X] — covers [Y and Z]».

> **Правило большого пальца:** Если вы объясняли что-то Claude больше двух раз в разных сессиях — это должно быть skill'ом.

## 4. Superpowers — плагин, который меняет всё

Если вы установите из этого руководства только один плагин — пусть будет этот.

Вот в чём дело: Claude *способен* на отличные инженерные практики — планирование перед кодингом, написание тестов сначала, методичная отладка. Но без направления он склонен сразу бросаться писать код. Мы все так делаем. Разница в том, что Superpowers делает эти практики **автоматическими и обязательными**, а не просто желательными.

```bash
claude plugin install superpowers@claude-plugins-official
```

### Что меняется после установки

Superpowers добавляет набор **процессных skills**, которые активируются исходя из того, что вы делаете:

| Когда вы... | Superpowers активирует... | Что значит, что Claude будет... |
|-------------|--------------------------|--------------------------------|
| Начинаете что-то новое | **Brainstorming** | Предлагать 2-3 подхода с trade-off'ами перед написанием любого кода |
| Готовы строить | **Writing Plans** | Создавать пошаговый план реализации для вашего одобрения |
| Строите | **Executing Plans** | Следовать плану шаг за шагом с отслеживанием прогресса |
| Пишете любую фичу или исправление | **Test-Driven Development** | Сначала писать падающий тест, потом реализацию |
| Ловите баг | **Systematic Debugging** | Формировать гипотезы, проверять методично, сужать причину |
| Заканчиваете кусок работы | **Code Review** | Проверять соответствие плану и стандартам проекта |
| Собираетесь сказать «готово» | **Verification** | Проводить финальные проверки перед объявлением завершения |
| Сталкиваетесь с независимыми задачами | **Parallel Agents** | Запускать несколько агентов одновременно |
| Начинаете работу над фичей | **Git Worktrees** | Изолировать работу в отдельной ветке worktree |
| Завершаете ветку | **Finishing a Branch** | Чисто финализировать ветку и подготовить PR |

### Почему это критически важно

**Без Superpowers** типичная сессия выглядит так:

```text
Вы: Implement user authentication with OAuth

Claude: Конечно! Давайте создам auth middleware...
        [пишет 200 строк кода]
        [понимает, что нужен другой подход]
        [переписывает половину]
        Готово! Я реализовал OAuth-аутентификацию.
        [нет тестов, нет плана, нет проверки]
```

**С Superpowers:**

```text
Вы: Implement user authentication with OAuth

Claude: [активируется Brainstorming]
        Вижу три подхода:
        1. Passport.js — зрелый, много стратегий, но тяжёлый
        2. Auth.js (NextAuth) — создан для Next.js, проще
        3. Кастомный с jose — лёгкий, полный контроль
        Вот trade-off'ы...

Вы: Вариант 2, идём с Auth.js

Claude: [активируется Planning]
        Вот план реализации:
        Шаг 1: Установить зависимости и настроить провайдеры
        Шаг 2: Создать auth API routes
        Шаг 3: Добавить session middleware
        Шаг 4: Защитить routes с auth guards
        ...

Вы: Хорошо, давай

Claude: [активируется Execution]
        Начинаем Шаг 1...
        [активируется TDD] Пишу тест для конфигурации OAuth-провайдера...
        [тест падает — отлично]
        [пишет реализацию, чтобы тест прошёл]
        [тест проходит]
        Шаг 1 завершён. Переходим к Шагу 2...
        ...
        [все шаги выполнены]
        [активируется Verification]
        Запускаю весь тестовый набор... Все проходят.
        Сверяю с планом... Все шаги покрыты.
        Готово.
```

Один и тот же запрос — совершенно разный результат. План ловит неверные повороты на раннем этапе. Тесты ловят регрессии. Проверка предотвращает сюрпризы «у меня работает».

### Воркфлоу с планированием устраняет главный пожиратель времени

Самый большой расход времени при AI-разработке — не медленные модели и не плохой код. Это **построение не того, что нужно**. Вы говорите «добавь аутентификацию», Claude интерпретирует это по-своему, вы хотели другое — и вы потратили 20 минут на код, который пойдёт в корзину.

Воркфлоу Superpowers «план → выполнение» устраняет это. Claude показывает план *до* написания кода. Вы его просматриваете, корректируете и только потом говорите «давай». 2 минуты на ревью плана экономят 20 минут на отмену неправильного подхода.

### Триггеры раздумий для критических решений

Когда вам нужно, чтобы Claude думал особенно тщательно — архитектурные решения, сложный рефакторинг, запутанная отладка — можно довести рассуждение до максимума:

- **`ultrathink`** — добавьте это слово в любой промпт для максимального рассуждения в этом конкретном запросе
- **`/effort high`** — установите постоянный уровень усилий для всей сессии
- **EXPLORE -> PLAN -> CODE -> COMMIT** — пусть Claude сначала читает файлы без написания кода, потом `ultrathink` над планом, критикует его на граничные случаи, *потом* выполняет

Особенно мощно в сочетании с воркфлоу планирования Superpowers. Дайте Claude провести брейнсторминг на обычном уровне, потом `ultrathink` над финальным планом перед выполнением.

## 5. Serena — навигация по коду, которая понимает код

Чаще всего, когда Claude нужно разобраться в вашем коде, он делает одно из двух: читает весь файл или запускает текстовый поиск. Оба варианта работают, но оба расточительны. Чтение 500-строчного файла ради одной функции сжигает контекст. Grep по имени класса находит определение, но упускает, что его *использует*.

Serena меняет это. Это языково-осведомлённый MCP-сервер, который понимает ваш код как **символы** — классы, функции, методы, переменные — а не просто текст. Он знает, что делает функция, где она определена и всё, что на неё ссылается.

### Как ощущается разница

| Старый способ | С Serena |
|---------------|----------|
| `grep -r "class UserService"` по всему проекту | `find_symbol("UserService")` — сразу туда |
| Читать весь файл, чтобы понять структуру | `get_symbols_overview("user-service.ts")` — только схема |
| Find-and-replace для переименования | `rename_symbol("oldName", "newName")` — безопасно, семантически |
| Вручную искать «кто это вызывает?» | `find_referencing_symbols("UserService")` — полный ответ |

Экономия токенов сама по себе значительная. Но настоящий выигрыш — точность: Serena находит *определение*, а не просто совпадения строк.

### Настройка для проекта

Два шага. Сначала создайте Serena-конфиг в проекте:

```yaml
# .serena/settings.yml
project_name: "my-project"
language_server:
  language: typescript  # or python, swift, go, etc.
  root_path: "src"
```

Затем скажите Claude активировать его. В проектном CLAUDE.md:

```markdown
## Serena (Semantic Code Analysis)
At session start, call `activate_project("my-project")`.
```

(Если вы настроили hook для старта сессии — это из [Раздела 11](#11-hooks-и-защита--автоматизация-и-безопасность) — это произойдёт автоматически.)

### Воркфлоу Serena

Самый эффективный способ исследовать код с Serena — паттерн **приближения**:

```text
list_dir                → Что находится в этой директории?
get_symbols_overview    → Какие символы есть в этом файле? (без тел — только имена и сигнатуры)
find_symbol             → Покажи полное тело этой одной функции
find_referencing_symbols → Кто это использует? (перед изменением)
```

И для редактирования:

```text
replace_symbol_body   → Точно переписать тело функции
insert_before_symbol  → Добавить что-то выше символа
insert_after_symbol   → Добавить что-то ниже символа
```

### Где Serena не поможет

Serena индексирует только **символы кода**. Она не видит:
- Конфиг-файлы (JSON, YAML, TOML)
- Markdown и документацию
- Строковые литералы, сообщения об ошибках, комментарии
- Некодовые файлы любого рода

Для этого нужен текстовый поиск — и это подводит нас к...

## 6. Ripgrep — когда текстовый поиск нужнее

Ripgrep (`rg`) — быстрый инструмент для текстового поиска, который Claude Code использует под капотом для своего инструмента `Grep`. По умолчанию Claude использует встроенную версию, но можно указать системный install для лучшей производительности.

### Быстрая настройка

```json
// ~/.claude/settings.json
{
  "env": {
    "USE_BUILTIN_RIPGREP": "0"
  }
}
```

```bash
# Установите, если ещё не стоит
brew install ripgrep        # macOS
sudo apt install ripgrep    # Ubuntu/Debian
```

### Serena vs Ripgrep — разделение труда

Эти два инструмента идеально дополняют друг друга. Объяснить Claude, когда использовать каждый из них — одна из самых ценных вещей, которые вы можете сделать (подробнее в [Разделе 7](#7-учим-claude-что-когда-использовать)):

| Берите Ripgrep когда нужно | Берите Serena когда нужно |
|---------------------------|--------------------------|
| Строковые литералы, сообщения ошибок | Определения функций/классов |
| Значения конфигов в нескольких файлах | Кто ссылается на этот символ? |
| Некодовые файлы (документы, JSON, YAML) | Тело конкретного метода |
| Regex-паттерны где угодно | Безопасное переименование символов |
| «Найти каждый файл, упоминающий X» | «Как выглядит API этого файла?» |

Ни один инструмент не заменяет другой. Вместе они покрывают всё.

## 7. Учим Claude что когда использовать

Это, пожалуй, самое ценное из всего руководства — и это просто таблица.

У Claude много доступных инструментов — Serena, Ripgrep, Glob, Context7, Graphiti, Edit, Read. Без направления он возвращается к своим обучающим значениям по умолчанию: читать целые файлы и делать grep по тексту. Это нормально, но это как использовать молоток для всего, когда у вас ещё есть отвёртка и дрель.

### Таблица

Добавьте это в ваш глобальный CLAUDE.md. Серьёзно, просто вставьте:

| Задача | Лучший инструмент | Почему |
|--------|-------------------|--------|
| Найти символ/класс/функцию | Serena `find_symbol` | Семантически, с пониманием языка |
| Понять структуру файла | Serena `get_symbols_overview` | Экономит токены — только схема |
| Проверить ссылки перед изменением | Serena `find_referencing_symbols` | Полный результат — ловит каждое использование |
| Поиск строковых литералов, значений конфигов, сообщений ошибок | `Grep` (ripgrep) | Текстовые паттерны, которые не являются символами кода |
| Поиск в некодовых файлах (markdown, JSON, YAML) | `Grep` (ripgrep) | Serena индексирует только код |
| Найти файлы по имени или паттерну | `Glob` | Быстрее любого поиска для обнаружения файлов |
| Редактировать тело функции/метода | Serena `replace_symbol_body` | Точная замена на уровне символа |
| Редактировать конфиг или некодовые файлы | `Edit` | Стандартное текстовое редактирование |
| Узнать текущее API библиотеки | Context7 | Актуальные доки — обучающие данные могут быть устаревшими |
| Вспомнить прошлое решение или паттерн | Graphiti `search_memory_facts` | Граф знаний между сессиями |

**Правило большого пальца:** Serena для символов кода. Grep для текста и некодовых файлов. Glob для файлов. Context7 для документации. Graphiti для памяти.

### Почему это важнее, чем кажется

Без этой таблицы типичная сессия Claude может:
1. Прочитать целый 400-строчный файл ради одной функции (40 токенов/строка = 16 000 потраченных токенов)
2. Сделать grep по имени класса и получить совпадения строк в комментариях, тестах и документации
3. Попытаться вспомнить, менялось ли API Zustand после даты обрезки обучающих данных

С таблицей:
1. `get_symbols_overview` → 200 токенов на схему всего файла, затем читает тело одной функции
2. `find_symbol` → сразу идёт к определению
3. Context7 → получает актуальную документацию

Та же задача, меньше токенов, лучше точность. За сессию это накапливается.

## 8. Graphiti — память, которая живёт между сессиями

Есть такой раздражающий паттерн: вы тратите 30 минут, объясняя Claude вашу архитектуру. Хорошо продвигаетесь. В следующей сессии объясняете всё снова. CLAUDE.md помогает — но он статичный, и вам нужно помнить его обновлять.

Graphiti работает иначе. Это **граф знаний**, в который Claude пишет во время сессий и из которого читает в начале. Со временем он накапливает паутину сущностей, связей и фактов о вашем проекте — с временными метками, чтобы вы знали, когда что-то было решено и актуально ли это.

### Что хранить в Graphiti (а что нет)

Думайте о Graphiti как о месте для **решений с контекстом** — того, что слишком детально или слишком изменчиво для CLAUDE.md:

**Да:**
- «Мы выбрали event sourcing для заказов, потому что нужен audit trail. Trade-off — более сложные чтения через проекции.»
- «Переписывание auth middleware обусловлено требованиями соответствия, а не техдолгом — при принятии решений по скоупу соответствие важнее удобства.»
- «React Query v5 требует оборачивания мутаций в startTransition — обнаружено при обновлении.»

**Нет:**
- Рутинные изменения кода (для этого есть git log)
- Секреты и учётные данные (никогда)
- Объёмные дампы кода (храните в коде, не в памяти)
- Всё, что уже есть в CLAUDE.md (не дублируйте)

### Как это работает на практике

Каждый проект получает `group_id` для изоляции знаний. Добавьте в проектный CLAUDE.md:

```markdown
## Memory (Graphiti Knowledge Graph)
When using Graphiti tools, always use `group_id="my-project"`.
```

**Начало сессии** — Claude загружает релевантный контекст:

```text
search_memory_facts(
  query="architecture decisions patterns",
  group_ids=["my-project"]
)
```

**Конец сессии** — Claude сохраняет то, что узнал:

```text
add_memory(
  group_id="my-project",
  content="Decided to use event sourcing for order processing
           because we need audit trail and temporal queries."
)
```

Самое приятное — оба действия можно автоматизировать через hooks (см. [Раздел 11](#11-hooks-и-защита--автоматизация-и-безопасность)), чтобы это *просто происходило* без ваших напоминаний.

## 9. Память — нужная информация в нужном месте

Claude Code имеет несколько систем памяти, и каждая делает что-то своё. Использовать неправильную — как класть чек в дневник: сохранено, но никогда не найдёте, когда понадобится.

### Шпаргалка

| Что храните | Куда | Почему |
|-------------|------|--------|
| Постоянные инструкции и правила | **CLAUDE.md** | Загружается всегда — вы пишете это |
| Быстро доступные факты (ID, имена, флаги статуса) | **MEMORY.md** | Автозагружается для проекта, легко просматривать |
| Паттерны, которые Claude обнаружил в работе | **Auto Memory** | Claude сам это пишет по мере работы |
| Решения с богатым контекстом (причины) | **Graphiti** | Поиск, временные метки, реляционность |
| Прогресс текущей задачи | **Tasks** | Переживает сжатие контекста |
| Глубокие тематические знания | **Skills** | Загружаются по требованию, не всегда |

### Auto Memory — Claude учит сам себя

Это отдельно от MEMORY.md, который пишете вы. Claude Code автоматически записывает заметки сам себе по мере работы — команды сборки, которые работали, инсайты из отладки, обнаруженная архитектура, исправления стиля кода, которые вы давали. Они накапливаются в `~/.claude/projects/<project>/memory/`.

Ключевое различие:
- **CLAUDE.md** = правила, которые пишете *вы*, загружаются каждую сессию
- **Auto Memory** = паттерны, которые *Claude* узнаёт, тоже загружаются каждую сессию
- Не дублируйте между ними — если Claude уже узнал что-то через auto memory, вам не нужно добавлять это в CLAUDE.md

### MEMORY.md на практике

Находится в `~/.claude/projects/<slug>/memory/MEMORY.md`, загружается автоматически в каждой сессии для этого проекта.

**Хороший контент для MEMORY.md:**
- «Supabase project ID: `abc123`»
- «Миграция на v2 auth: завершена по состоянию на 2026-01-15»
- «Имена edge functions: `process-payment`, `send-notification`»

**Плохой контент для MEMORY.md:**
- Паттерны кода (читайте кодовую базу)
- Сводки git-истории (запустите `git log`)
- Рецепты исправлений (исправление есть в коде)
- Всё, что уже есть в CLAUDE.md

**Держите файл до 200 строк.** Для глубоких тем ссылайтесь на отдельные файлы:

```markdown
<!-- MEMORY.md -->
- [Supabase setup](supabase.md) — project ID, edge functions, RLS policies
- [Localization](localization.md) — supported languages, string key format
```

## 10. Context7 — документация, которая актуальна

Обучающие данные Claude имеют дату отсечения. Библиотеки выпускают обновления после этой даты. Значит, Claude может уверенно писать код, используя API, которое изменилось два месяца назад. Context7 исправляет это, получая **живую документацию** в момент, когда Claude её нужна.

```bash
claude plugin install context7@claude-plugins-official
```

### Когда это важно

- Написание интеграционного кода для любой библиотеки (React, Express, Prisma, Tailwind — для всех)
- Обновление или миграция между версиями библиотек
- Отладка поведения, специфичного для библиотеки
- Работа с библиотекой, в которой Claude явно неуверен

### Научите Claude обращаться к ней

Добавьте в глобальный CLAUDE.md:

```markdown
## Context7
Call Context7 before writing integration code for any library.
Training data may be outdated — always verify current API.
```

### Когда не нужна

Context7 для документации по библиотекам. Не используйте для общих концепций программирования, рефакторинга вашего собственного кода, бизнес-логики или code review.

## 11. Hooks и защита — автоматизация и безопасность

Hooks служат двум целям: **автоматизация** (чтобы вы не забывали делать вещи) и **защита** (чтобы Claude не делал то, чего не должен). Комбинация Claude Code hooks и git hooks создаёт надёжную защитную сетку.

### Старт сессии: загружать всё автоматически

Настройте hook, который запускается в начале каждой сессии:

```json
// ~/.claude/settings.json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/session-start.sh",
            "statusMessage": "Loading project context"
          }
        ]
      }
    ]
  }
}
```

Скрипт hook'а читает ваш проектный CLAUDE.md, извлекает Graphiti `group_id` и имя проекта Serena и встраивает напоминания:

```bash
#!/bin/bash
INPUT=$(cat)
CWD=$(echo "$INPUT" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('cwd',''))" 2>/dev/null)

GROUP_ID=""
SERENA_NAME=""

if [ -f "$CWD/CLAUDE.md" ]; then
  GROUP_ID=$(sed -n 's/.*group_id="\([^"]*\)".*/\1/p' "$CWD/CLAUDE.md" | head -1)
  SERENA_NAME=$(sed -n 's/.*activate_project("\([^"]*\)".*/\1/p' "$CWD/CLAUDE.md" | head -1)
fi

MSGS="Action required: Read ~/.claude/CLAUDE.md for global instructions.\n"
[ -n "$SERENA_NAME" ] && MSGS="${MSGS}Action required: activate_project(\"$SERENA_NAME\")\n"
[ -n "$GROUP_ID" ] && MSGS="${MSGS}Action required: search Graphiti with group_id=$GROUP_ID\n"
echo -e "$MSGS"
```

Теперь каждая сессия стартует полностью подготовленной — никаких ручных напоминаний.

### Конец сессии: сохраняйте то, что узнали

Stop hook запускается когда Claude собирается завершить работу. Используйте его, чтобы не потерять несохранённые знания:

```json
{
  "stopHooks": [
    {
      "type": "prompt",
      "prompt": "Before stopping, check if any architectural decisions or bug root causes were discovered. If so, save to Graphiti. Check if MEMORY.md needs new identifiers or status updates.",
      "model": "haiku"
    }
  ]
}
```

> **Почему Haiku?** Stop hook — это быстрый аудит, ему не нужны тяжёлые рассуждения. Haiku делает это быстро и дёшево.

### Типы hooks: не только shell-скрипты

Hooks бывают трёх видов, каждый для своего уровня проверки:

| Тип | Что делает | Для чего лучше всего |
|-----|-----------|---------------------|
| `command` | Запускает shell-скрипт, проверяет код выхода | Линтинг, форматирование, уведомления |
| `prompt` | Отправляет промпт модели Claude для одноразовой проверки | Лёгкое ревью, проверки безопасности |
| `agent` | Запускает subagent с полным доступом к инструментам (до 50 ходов) | Глубокая проверка, проверки в нескольких файлах |

`prompt` hook отлично подходит для проверок «безопасен ли этот код?» без накладных расходов полного subagent. `agent` hook может читать файлы, запускать поиск и проверять условия по всей кодовой базе, прежде чем разрешить вызов инструмента.

### Git hooks: второй уровень защиты

Не полагайтесь только на Claude Code hooks. Git hooks ловят проблемы при коммите, независимо от того, как был написан код:

- **pre-commit** — запускать линтеры, форматтеры, проверки типов
- **commit-msg** — соблюдать формат conventional commit
- **pre-push** — запускать тесты перед push

Две системы hooks дополняют друг друга:
- **Claude Code hooks** ловят проблемы *во время* сессии (до того, как Claude запишет файл)
- **Git hooks** ловят проблемы *при коммите* (последний барьер перед попаданием в историю)

Вместе они создают эшелонированную оборону. Claude Code hooks предотвращают запись плохих паттернов. Git hooks не дают ничему, что просочилось, попасть в коммит.

### Типичные паттерны hooks

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Review this file change for security issues (injection, XSS, exposed secrets). Return {\"ok\": true} if safe, {\"ok\": false, \"reason\": \"...\"} if not.",
            "model": "haiku"
          }
        ]
      }
    ]
  }
}
```

### Результат

С обеими системами hooks ваш воркфлоу имеет несколько контрольных точек:
- **Старт сессии:** инструменты активированы, память загружена, контекст готов
- **Во время работы:** проверки безопасности при правках, форматирование при сохранении
- **При коммите:** линтинг, тесты, формат сообщения
- **Конец сессии:** решения сохранены, память обновлена, ничего не потеряно

## 12. Права и безопасность

Система прав Claude Code контролирует, что Claude может делать без вашего разрешения. Правильная настройка позволяет Claude быстро работать с безопасными операциями, сохраняя ограничения на всё опасное.

### Три уровня

```json
{
  "permissions": {
    "allow": [
      "Read(**)",                          // Читать всё
      "Edit(**)",                          // Редактировать всё
      "Bash(git:*)",                       // Все git-команды
      "Bash(npm:*)",                       // Все npm-команды
      "mcp__plugin_serena_serena__*"       // Все инструменты Serena
    ],
    "deny": [
      "Bash(sudo *)",                      // Никогда sudo
      "Bash(git push --force *)",          // Никогда force push
      "Read(.env)",                        // Никогда не читать секреты
      "Read(**/*.key)",                    // Никогда не читать key-файлы
      "Read(~/.ssh/**)",                   // Никогда не читать SSH-ключи
      "Read(~/.aws/**)",                   // Никогда не читать AWS-credentials
      "Read(~/.kube/**)",                  // Никогда не читать kube config
      "Read(~/.npmrc)",                    // Никогда не читать npm auth
      "Read(~/.git-credentials)",          // Никогда не читать git credentials
      "Edit(~/.bashrc)",                   // Никогда не менять shell config
      "Edit(~/.zshrc)"                     // Никогда не менять shell config
    ],
    "ask": [
      "Bash(curl * | bash*)"              // Спрашивать перед pipe-to-bash
    ]
  }
}
```

### Принципы

1. **Щедро с чтением** — Claude работает лучше, когда может свободно исследовать
2. **Строго с деструктивными операциями** — force push, дропы схем, `rm -rf` на системных путях
3. **Всегда запрещать доступ к секретам** — `.env`, credentials, ключи, PEM-файлы, SSH, AWS, kube configs
4. **Wildcard для MCP-инструментов** — `mcp__plugin_serena_serena__*` даёт доступ ко всем инструментам Serena сразу
5. **`ask` для серых зон** — то, что иногда нормально, но иногда опасно

### Защита от вредоносных MCP-серверов в клонированных репо

Репозитории могут содержать файлы `.mcp.json`, которые автоматически активируют MCP-серверы при открытии. Это риск цепочки поставок — вредоносное репо может внедрить инструменты. Отключите автоактивацию:

```json
{
  "enableAllProjectMcpServers": false
}
```

С такой настройкой MCP-серверы на уровне проекта требуют явного включения.

### Режим bypass (для продвинутых)

Если вы доверяете своему deny-списку и не хотите запросов разрешений на всё остальное:

```json
{
  "permissions": {
    "defaultMode": "bypassPermissions"
  },
  "skipDangerousModePermissionPrompt": true
}
```

Это значит, что Claude остановится только на том, что явно запрещено. Быстро, но требует хорошо настроенных deny-правил. Не включайте, пока ваши deny-правила не будут надёжными.

## 13. Subagents и команды агентов

Claude Code может запускать меньшие экземпляры Claude для обработки подзадач. Это одна из самых мощных функций, когда выходишь за рамки настроек по умолчанию.

### Выбор модели: правильный мозг для правильной задачи

| Модель | Лучше всего для | Trade-off |
|--------|----------------|-----------|
| **Haiku** | Поиск файлов, быстрые поиски, исследование | Быстро и дёшево — но может пропустить нюансы |
| **Sonnet** | Реализация, тесты, code review, стандартное QA | Лучший баланс — ваш ежедневный инструмент |
| **Opus** | Архитектурные решения, security-critical код, сложные рефакторы | Глубочайшее рассуждение — стоит для высокорисковой работы |

Добавьте в глобальный CLAUDE.md:

```markdown
## Subagents
- haiku — exploration, quick lookups
- sonnet — implementation, tests, code review, standard QA
- opus — architecture, security-critical, complex refactors
```

### Кастомные определения subagent

Помимо встроенных агентов, можно определить собственных специализированных агентов в `.claude/agents/`. Они переиспользуются между сессиями и могут накапливать знания со временем.

Пример — code reviewer, знающий проект:

```markdown
<!-- .claude/agents/code-reviewer.md -->
---
name: code-reviewer
description: Reviews code changes for bugs, security issues, and adherence to project conventions
model: sonnet
memory: project
tools:
  - Read
  - Grep
  - Glob
---

You are a code reviewer for this project. Review changes for:
- Logic errors and edge cases
- Security vulnerabilities
- Adherence to project conventions in CLAUDE.md
- Test coverage gaps

Be specific. Cite line numbers. Suggest fixes, don't just flag problems.
```

Настройка `memory: project` означает, что этот агент накапливает знания о кодовой базе в `~/.claude/agent-memory/code-reviewer/` между сессиями. Чем больше вы его используете, тем лучше он понимает вашу кодовую базу.

Определение агентов под ваши конкретные потребности — это меняет игру: security reviewer, test writer, architecture advisor. У каждого своя модель, инструменты и память.

### Команды агентов

> *Я сам ещё не использовал Agent Teams, но стоит знать об этом для сложных проектов.*

Agent Teams выходят за рамки простых subagents. Вместо однонаправленного делегирования родитель-потомок, у команд есть:
- **Общие списки задач** с отслеживанием зависимостей
- **Peer-to-peer общение** между агентами
- **Параллельное выполнение** — каждый участник команды — полноценный экземпляр Claude Code

Думайте об этом как о переходе от «одного разработчика с ассистентами» к «небольшой команде, работающей вместе».

Включите:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

Лучшие сценарии: параллельный code review (security + performance + test reviewer), разработка независимых модулей, кросс-слойные изменения где фронтенд и бэкенд должны двигаться вместе.

**Рекомендации по размеру:** 3-5 участников, 5-6 задач на участника. Каждый участник сжигает свои токены, так что это мощный инструмент — используйте, когда параллелизм оправдывает затраты.

## 14. Управление контекстом

Это, пожалуй, самый важный операционный вывод в этом руководстве: **Claude работает лучше всего, когда контекстное окно не превышает 200К токенов.** Не 500К, не 1М — именно до 200К. Да, окно *поддерживает* 1М, но качество деградирует по мере заполнения. Представьте это как RAM: в компьютере 64 ГБ, но если используется 60 ГБ — всё замедляется.

Из этого следует прямое практическое следствие: **compact and continue почти всегда лучше, чем начинать новую сессию** для схожей работы. Когда вы делаете `/compact`, Claude сохраняет ключевые решения и контекст, отбрасывая шум. Сжатая сессия с 50К сфокусированного контекста превосходит свежую, где Claude нужно перечитывать всё с нуля — и она *значительно* превосходит раздутую сессию на 500К, где важные инструкции погребены под старыми выводами инструментов.

### Ключевые команды

| Команда | Что делает | Когда использовать |
|---------|-----------|-------------------|
| `/compact` | Сжимает историю разговора, сохраняя ключевой контекст | Когда сессия ощущается медленной или вы переключаетесь на подзадачу |
| `/compact Focus on the API changes` | Направленное сжатие — говорит Claude, что сохранить | Когда нужно, чтобы конкретный контекст выжил |
| `/clear` | Полностью стирает историю разговора | Между несвязанными задачами — самая недооценённая команда |
| `/btw` | Задать боковой вопрос, который не попадёт в историю | Быстрые поиски, которые не должны раздувать контекст |
| `/context` | Показывает разбивку токенов по категориям | Диагностика того, что съедает контекст |
| `Ctrl+B` | Перевести запущенный subagent в фон | Когда subagent тормозит и вы хотите сделать что-то другое |

### Compact vs clear vs new session

Это решение вы будете принимать десятки раз. Вот правило:

| Ситуация | Лучшее действие | Почему |
|----------|----------------|--------|
| Переключаетесь на **несвязанную** работу | `/clear` | Старый контекст — чистый шум для новой задачи |
| Продолжаете **схожую** работу после завершённого куска | `/compact` | Оставить решения, сбросить выводы инструментов |
| Сессия тормозит или Claude забывает инструкции | `/compact` с указаниями | Сбросить шум, сохранив важное |
| Начинаете совершенно **другой проект** | Новая сессия | Другой CLAUDE.md, другой проект Serena, всё другое |

Ключевой вывод: **`/compact` + продолжение бьёт новую сессию**, когда работа связана. Сжатая сессия сохраняет архитектурные решения, согласованные паттерны и прогресс задачи. Новая сессия стартует холодной и должна заново всё это обнаружить.

### Практические привычки

**Используйте `/clear` агрессивно между несвязанными задачами.** Закончили отлаживать API-эндпоинт? `/clear` перед началом работы над фронтендом. Цена перечитывания нескольких файлов — ничто по сравнению с тащиванием 100К нерелевантного контекста отладки.

**Compact проактивно, а не реактивно.** Не ждите, пока Claude начнёт путаться. После завершения крупного шага сделайте `/compact`, чтобы контекст оставался лёгким. Думайте об этом как об уборке стола между задачами — маленькая привычка, которая накапливается.

**Кастомные инструкции для `/compact` работают.** Вместо просто `/compact` попробуйте:

```text
/compact Keep the architectural decisions and test commands,
         summarize the debugging session
```

Это направляет, что переживёт сжатие. Полезно, когда вы собираетесь сменить фокус, но хотите сохранить конкретные решения в контексте.

**`/btw` для быстрых поисков.** Нужно проверить сигнатуру функции или значение конфига, но не хотите загромождать разговор? `/btw what's the return type of getUserById?` — ответ появляется в свёртываемом оверлее, никогда не попадает в историю.

**Subagents — ваша лучшая защита контекста.** Делегируйте исследовательскую работу subagents. Они работают в собственном контексте, возвращают только ответ и держат основную сессию чистой. «Используй subagent для исследования тестовых сбоев в модуле auth» лучше, чем исследовать напрямую и заполнять сессию 50 прочитанными файлами.

### Что переживает сжатие

- CLAUDE.md: перечитывается с диска при каждом сжатии — всегда выживает
- Auto Memory / MEMORY.md: всегда загружается — всегда выживает
- Tasks: переживают сжатие — вот почему они важны
- Инструкции из разговора: **НЕ выживают**, если не записаны в CLAUDE.md

Если инструкция, которую вы дали Claude, исчезла после сжатия — она была только в разговоре. Запишите в CLAUDE.md или в skill, если она должна быть постоянной.

## 15. Git Worktrees — параллельные сессии

Claude Code *создан* для параллелизма. Это не запоздалая мысль — это ключевой архитектурный принцип. Subagents работают параллельно внутри сессии. Несколько сессий работают параллельно между worktrees. И вы можете комбинировать оба подхода: основная сессия оркестрирует subagents, пока другие сессии работают независимо в своих worktrees.

Когда вы это усваиваете, вы перестаёте думать о Claude как об «одном ассистенте, делающем одно дело», и начинаете думать о нём как о «команде, которую я могу развернуть по всей кодовой базе».

### Идея

Git worktree — это отдельный чекаут того же репо в другой директории, на своей ветке. Каждая сессия Claude работает в своём worktree — разные файлы, разная ветка, никаких конфликтов.

### Встроенная поддержка

Claude Code имеет первоклассную поддержку worktrees, а плагин Superpowers включает skill **Using Git Worktrees**, который управляет воркфлоу:

```bash
# Начать сессию в новом worktree
claude --worktree feature-auth
```

Это создаёт `.claude/worktrees/feature-auth/` с выделенной веткой.

### Паттерн для продвинутых

Высокопроизводительные команды запускают 5-15 сессий Claude одновременно — одни во вкладках терминала, другие в веб-приложении — каждая в своём worktree. Фронтенд в одном, бэкенд в другом, тесты в третьем. Все работают параллельно без конфликтов, и каждая сессия открывает свой PR по завершении.

### Два уровня параллелизма

Думайте об этом как о двух взаимодополняющих стратегиях:

**Внутри сессии — subagents.** Claude отправляет subagents Haiku/Sonnet/Opus для параллельной обработки подзадач. Исследуйте три подхода одновременно. Запускайте code review и тесты в одно время. Изучайте кодовую базу в одном subagent, пока планируете в другом. Это сохраняет контекст основной сессии чистым, при этом работа делается быстрее.

**Между сессиями — worktrees.** Несколько полноценных сессий Claude, каждая в своём worktree, работающих над независимыми задачами. Так вы распараллеливаете на уровне фич — аутентификация в одной сессии, API-эндпоинты в другой, миграции базы данных в третьей.

Комбинация — это то, где настоящая сила. Каждая сессия worktree сама *может запускать* subagents. В итоге вы получаете дерево параллельной работы, координация которой заняла бы у команды людей дни.

### Советы по настройке

**Добавьте директорию worktrees в gitignore:**

```gitignore
.claude/worktrees/
```

**Копируйте env-файлы в новые worktrees** через `.worktreeinclude`:

```text
# .worktreeinclude — gitignored files to copy into each new worktree
.env.local
.env.development
```

Копируются только gitignored файлы — отслеживаемые файлы приходят из git checkout автоматически.

### Worktrees для subagents

Можно также давать subagents их собственные worktrees для по-настоящему изолированной параллельной работы:

```markdown
<!-- .claude/agents/my-agent.md -->
---
isolation: worktree
---
```

Worktree автоматически очищается, если агент не делает изменений.

## 16. Git и code review

### Conventional commits

Определите формат один раз в глобальном CLAUDE.md и больше никогда об этом не думайте:

```markdown
## Git
Commits: <type>(<scope>): <description> (50 char max subject)
Body: blank line + bullet list, 72 char wrap
Types: feat fix docs style refactor perf test build ci chore revert
Rules: Never commit to main. Never force push. Never skip hooks.
git commit is auto-approved. git push requires confirmation.
```

### Code review: используйте оба подхода

Лучшая настройка code review использует **два взаимодополняющих подхода**:

**1. [CodeRabbit](https://www.coderabbit.ai/) — облачный AI review на PR**

CodeRabbit автоматически проверяет ваши pull requests на GitHub. Он ловит проблемы со свежим взглядом — он не видел контекста вашей сессии, поэтому находит то, что вы и Claude упустили.

```bash
claude plugin install coderabbit@claude-plugins-official
```

```text
/coderabbit:review    — Run CodeRabbit review on current changes
```

**Воркфлоу для CodeRabbit reviews:**
1. Разбирайте все комментарии — включая «nice to have» предложения
2. Отмечайте каждый тред как resolved после исправления
3. Отправьте `@coderabbitai resolve` для очистки оставшихся комментариев
4. Отправьте `@coderabbitai full review` для нового прохода
5. CodeRabbit переходит в APPROVED когда все треды чистые

**2. Локальные subagent reviews — немедленная, контекстная обратная связь**

Claude Code может запускать review subagents, которые проверяют вашу работу на соответствие плану и стандартам проекта прямо в вашей сессии. Плагин Superpowers делает это автоматически после завершения основных этапов реализации.

Можно также запустить вручную:

```text
/code-review
```

Или определить кастомного review агента (см. [Раздел 13](#13-subagents-и-команды-агентов)), который накапливает знания о вашей кодовой базе со временем.

**Почему оба?** Локальные reviews быстрые и контекстуальные — они знают план и сразу ловят отклонения. CodeRabbit reviews свежие и независимые — они ловят проблемы, которые упустит предвзятость сессии. Вместе они дают эшелонированную защиту.

## 17. Управление сессиями

Сессии в Claude Code — не одноразовые: их можно возобновлять, называть и переключаться между ними. Это важно для многодневной работы и когда нужно переключать контекст.

### Ключевые команды

| Команда | Что делает |
|---------|-----------|
| `claude --continue` | Продолжить последнюю сессию |
| `claude --resume` | Открыть выборщик сессий для выбора нужной |
| `claude -n auth-refactor` | Начать именованную сессию (легче найти потом) |
| `/rename auth-refactor` | Переименовать текущую сессию |
| `claude --from-pr 123` | Продолжить сессию, связанную с конкретным PR |

### Горячие клавиши в выборщике сессий

В пикере `/resume`:
- **`P`** — предпросмотр сессии
- **`R`** — переименовать
- **`B`** — фильтровать по текущей git-ветке
- **`A`** — переключать между текущей директорией и всеми проектами

### Префикс `!`

Нужно выполнить быструю shell-команду без лишнего roundtrip инструмента? Добавьте префикс `!`:

```text
! git status
! npm test
! ls -la src/
```

Вывод попадает прямо в контекст разговора. Быстрее, чем просить Claude выполнить это, и иногда этого достаточно.

## 18. Status Line — что происходит прямо сейчас

Claude Code поддерживает кастомную строку статуса внизу терминала, показывающую информацию о сессии в реальном времени. Настройте её с помощью shell-скрипта `~/.claude/statusline.sh`.

Хорошая строка статуса показывает:
- **Использование контекста** — насколько заполнено контекстное окно (зелёный < 50%, жёлтый 50-79%, красный 80%+)
- **Стоимость сессии** — сколько вы потратили в этой сессии
- **Прошедшее время** — как долго вы работаете
- **Процент попаданий в кэш промптов** — насколько эффективно переиспользуется контекст

Это даёт моментальное понимание, когда делать `/compact` или `/clear`, и помогает развить интуицию о том, что дорого, а что дёшево.

## 19. Отслеживание задач — не теряйте нить

Когда разговор становится длинным, Claude Code сжимает старые сообщения, чтобы оставаться в пределах контекста. Обычно это нормально — но это означает, что Claude может забыть, на каком шаге многоэтапного плана он находился.

Задачи переживают это сжатие. Они всегда видны, даже после compaction.

```text
TaskCreate   — Create a task
TaskUpdate   — Mark in_progress, completed, or blocked
TaskList     — See all current tasks
```

**Практика:** Для любой работы с более чем 2-3 шагами — создавайте задачи заранее. Отмечайте каждую как `in_progress` когда начинаете, `completed` когда закончили — сразу, не пакетом в конце.

> Задачи для текущей сессии. Для информации, которая должна сохраняться между сессиями, используйте MEMORY.md или Graphiti.

## 20. Типичные ошибки — чего избегать

Это ошибки, которые больше всего тратят время при работе с Claude Code. Каждая легко возникает и легко исправляется, когда знаешь, что искать.

### Сессия-кухонная-раковина

**Паттерн:** Вы отлаживаете API-эндпоинт, потом исправляете CSS-баг, потом пишете миграцию, потом рефакторите тест — всё в одной сессии. В конце Claude тащит 500К смешанного контекста и его ответы становятся расплывчатыми.

**Исправление:** `/clear` между несвязанными задачами. Кажется расточительным («У меня только что был весь этот контекст!»), но чистый старт с правильным контекстом всегда бьёт раздутую сессию.

### Цикл исправлений

**Паттерн:** Claude делает что-то не так. Вы исправляете. Он делает снова не так, немного иначе. Вы исправляете снова. Через три круга вы раздражены, а Claude запутан — каждое исправление добавляло конфликтующий контекст.

**Исправление:** После двух неудавшихся исправлений одной и той же проблемы — остановитесь. `/clear` и напишите лучший начальный промпт. Проблема не в понимании Claude — в том, что исходная инструкция была недостаточно ясной, и каждое исправление делает контекст более шумным.

### Перегруженный CLAUDE.md

**Паттерн:** Ваш CLAUDE.md — 800 строк детальных правил. Claude игнорирует половину из них, потому что важные правила погребены в шуме. Вы добавляете больше правил, чтобы компенсировать. Становится хуже.

**Исправление:** Каждая строка в CLAUDE.md должна проходить проверку: *если убрать это правило, Claude совершит конкретную ошибку?* Если Claude и так делает правильно без правила — удалите правило. Целевой объём ~100-200 строк. Глубокие знания переносите в skills с загрузкой по требованию.

### Разрыв доверия и проверки

**Паттерн:** Claude пишет правдоподобно выглядящий код. Вы беглым взглядом проверяете, говорите «выглядит хорошо» и двигаетесь дальше. Позже обнаруживаете, что он не обрабатывает граничные случаи или незаметно сломал что-то ещё.

**Исправление:** Всегда указывайте критерии проверки — тесты для запуска, скрипты для выполнения, конкретное поведение для проверки. Skill верификации Superpowers делает это автоматически, но можно просто сказать: «Прежде чем закончим, запусти тест-сьют и покажи вывод.»

### Бесконечное исследование

**Паттерн:** Вы просите Claude «исследовать проблемы с производительностью». Claude читает 80 файлов, запускает 30 поисков и заполняет ваш контекст исчерпывающим анализом, который вам не был нужен.

**Исправление:** Сужайте запросы. «Проверь, вызывает ли N+1 запрос в getUserOrders торможение» вместо «исследуй производительность». Или делегируйте subagent: «Используй subagent для исследования проблем производительности в модуле orders» — исследование происходит в отдельном контексте и возвращает только выводы.

### Ложь «добавлю тесты потом»

**Паттерн:** Claude реализует фичу без тестов. Вы планируете добавить их позже. Вы никогда не добавляете. Следующее изменение ломает фичу незаметно.

**Исправление:** Воркфлоу TDD Superpowers обрабатывает это автоматически — тесты сначала, а не последними. Если вы не используете Superpowers — сделайте это правилом в CLAUDE.md: «Write failing tests before implementation. No exceptions.»

## 21. Синхронизация конфигов — одинаковая настройка везде

Если вы разрабатываете больше чем на одной машине — скажем, рабочий ноутбук и личный — вы хотите, чтобы ваша настройка Claude Code была одинаковой на обеих. Тот же CLAUDE.md, те же hooks, те же права, те же плагины.

### cc-config-sync

[cc-config-sync](https://www.npmjs.com/package/cc-config-sync) — CLI-инструмент, который хранит конфиг Claude Code в git-репо и синхронизирует между машинами:

```bash
npm install -g cc-config-sync

cc-config-sync pull        # Захватить конфиг текущей машины в репо
cc-config-sync push --yes  # Применить конфиг репо к текущей машине
cc-config-sync status      # Проверить, что различается
cc-config-sync push --dry-run  # Предпросмотр без записи
```

### Что синхронизируется

- `~/.claude/CLAUDE.md` (глобальные инструкции)
- `~/.claude/settings.json` (права, hooks, плагины)
- `~/.claude/hooks/*.sh` (hook-скрипты)
- `~/.claude/plugins/installed_plugins.json`
- CLAUDE.md, настройки и файлы памяти для каждого проекта

### Одна вещь, на которую стоит обратить внимание

`settings.local.json` содержит машино-специфичные переопределения — например, пути, которые различаются между машинами. Проверяйте эти файлы перед push между окружениями. То, что работает на одной машине, может не подходить для другой.

## С чего начать

Если вы настраиваете всё с нуля — вот порядок, дающий максимум ценности быстрее всего:

1. **Установите основные плагины** — superpowers, claude-md-management, skill-creator, context7, serena, code-review, coderabbit
2. **Сгенерируйте первый CLAUDE.md** — используйте `/init` для scaffold, потом доработайте с `/claude-md-improver`
3. **Добавьте таблицу выбора инструментов** в глобальный CLAUDE.md — эта одна таблица меняет то, как работает Claude
4. **Настройте права** — читать разрешайте щедро, запрещайте опасное и секреты
5. **Настройте hooks** — старт сессии для автоматической активации инструментов, конец для сохранения памяти
6. **Напишите первый проектный CLAUDE.md** — архитектура, конвенции, ключевые файлы, команды сборки
7. **Настройте Serena** для главного проекта — экономия токенов начнётся сразу
8. **Настройте Graphiti** для постоянной памяти — решения перестанут теряться между сессиями
9. **Создайте первый skill** — возьмите что-то, что вы постоянно объясняете, и сделайте это on-demand
10. **Доверьтесь Superpowers** — позвольте ему управлять воркфлоу планирования, TDD и верификации

Каждый шаг строится на предыдущих. Не нужно делать всё сразу — даже шаги 1-5 дадут заметную разницу.
